## 1. Decomposable vs Non-Decomposable Statistics

### Taxonomy (Gray et al. 1996)

Gray et al. ("Data Cube: A Relational Aggregation Operator", ICDE 1996) established the canonical three-way classification of aggregate functions:

| Category | Definition | Examples |
|----------|-----------|----------|
| **Distributive** | `f(X ⊎ Y) = g(f(X), f(Y))` where `g` is a fixed binary function | count, sum, min, max |
| **Algebraic** | Computable from a bounded-size tuple of distributive aggregates | mean (= sum/count), variance (via count + mean + M2) |
| **Holistic** | No bounded-size partial state suffices for exact computation | median, mode, percentiles, trimmed mean, rank statistics |

This classification maps directly onto algebraic structure:

- **Distributive** = monoid homomorphism from the free commutative monoid on observations to a result monoid.
- **Algebraic** = monoid homomorphism into a *product* of monoids, followed by a finalization function.
- **Holistic** = no finite-dimensional monoid homomorphism exists.

### Fully Decomposable (Distributive)

These statistics are commutative monoid homomorphisms:

| Statistic | Carrier | Identity | Operation | Finalize |
|-----------|---------|----------|-----------|----------|
| count | N | 0 | + | id |
| sum | (T, +) | 0 | + | id |
| min | T ∪ {+∞} | +∞ | min | id |
| max | T ∪ {-∞} | -∞ | max | id |

### Algebraic (Decomposable via Tuple)

These require a fixed-size intermediate state but are still mergeable:

| Statistic | State | Merge | Finalize |
|-----------|-------|-------|----------|
| mean | (count, sum) | pointwise add | sum / count |
| variance | (count, mean, M2) | Chan-Golub-LeVeque merge | M2 / (count - 1) |
| standard deviation | (count, mean, M2) | Chan-Golub-LeVeque merge | sqrt(M2 / (count - 1)) |

### Holistic (Non-Decomposable)

These cannot be expressed as monoid homomorphisms with bounded state:

- **Median / percentiles**: The median of `X ⊎ Y` cannot be computed from the medians of `X` and `Y` alone. Counterexample: `median({1,2,3}) = 2`, `median({4,5,6}) = 5`, but `median({1,2,3,4,5,6}) = 3.5` — no function of `(2, 5)` alone can produce `3.5` in general.
- **Mode**: The most frequent element of a union depends on the full frequency distribution, not just the per-partition modes.
- **Trimmed mean**: Requires knowing which elements fall in the trimmed range across the full dataset.
- **Rank statistics** (arbitrary order statistics): Equivalent to the percentile problem.

**Space lower bound** (Munro & Paterson 1980): Computing the exact median of a stream of `n` elements in `p` passes requires `Ω(n^(1/p))` space. For a single pass, this is `Ω(n)` — i.e., you must store (essentially) the entire dataset.

### Approximately Decomposable (Sketch Structures)

These trade exactness for bounded state via approximate monoid homomorphisms:

| Structure | Target Statistic | Error Guarantee | Mergeable? | Reference |
|-----------|-----------------|-----------------|------------|-----------|
| **t-digest** (Dunning & Ertl 2019) | Quantiles | Rank-based, tighter at tails | Yes (approximate) | arXiv:1902.04023 |
| **DDSketch** (Masson et al. 2019) | Quantiles | Relative error (e.g., 1%) | Yes (exact merge) | VLDB 2019 |
| **HdrHistogram** (Tene) | Quantiles / latency | Configurable significant digits | Yes (exact merge) | github.com/HdrHistogram |
| **Count-Min Sketch** (Cormode & Muthukrishnan 2005) | Frequencies | Additive error ε·‖a‖₁ | Yes (exact merge) | J. Algorithms 2005 |

DDSketch is notable: it is the first quantile sketch that is *fully mergeable* (merge is exact — no error accumulation) while providing *relative* error guarantees. The merge operation is pointwise addition of bucket counts, which is a genuine commutative monoid operation.

## 2. Algebraic Structure

### Product Monoid Construction

Each distributive statistic `(S, e, ⊕)` is a commutative monoid. The product of commutative monoids is itself a commutative monoid under componentwise operation:

```
(S₁ × S₂ × ... × Sₖ,  (e₁, e₂, ..., eₖ),  ⊕₁ × ⊕₂ × ... × ⊕ₖ)
```

Therefore the tuple `(count, sum, min, max)` forms a commutative monoid:

```
State = (N, T, T ∪ {+∞}, T ∪ {-∞})
Identity = (0, zero, +∞, -∞)
Merge = (n₁ + n₂, s₁ + s₂, min(lo₁, lo₂), max(hi₁, hi₂))
```

This is exactly the product monoid `N_+ × (T,+) × (T,min) × (T,max)`.

### Welford / Chan-Golub-LeVeque as a Monoid

Chan, Golub & LeVeque (1979/1983) showed that the triple `(count, mean, M2)` supports a parallel merge:

```
Given A = (nₐ, μₐ, M2ₐ) and B = (n_b, μ_b, M2_b):

δ  = μ_b - μₐ
n  = nₐ + n_b
μ  = (nₐ · μₐ + n_b · μ_b) / n
M2 = M2ₐ + M2_b + δ² · nₐ · n_b / n
```

**Claim**: This merge is associative and has identity `(0, 0, 0)`.

**Proof sketch of associativity**: The M2 value equals `Σᵢ (xᵢ - μ)²` for the combined dataset. Since this quantity is uniquely determined by the multiset of observations, and multiset union is associative, the merge of any partition of the multiset into sub-partitions must yield the same result regardless of association order. The merge formula is simply the unique function that computes the combined `(count, mean, M2)` from two partial summaries — its associativity follows from the associativity of multiset union.

**Commutativity**: The merged `(n, μ, M2)` is symmetric in the two inputs (both contribute to the weighted mean and the cross-term `δ² · nₐ · n_b / n` is symmetric).

Therefore `(count, mean, M2)` with the CGL merge is a **commutative monoid**. The finalization step `variance = M2 / (n - 1)` (or `M2 / n` for population variance) is *not* part of the monoid — it is a post-processing projection. The algebraic aggregate is the monoid homomorphism into `(count, mean, M2)` followed by this projection.

### Formal Statement

Let `M(X)` be the free commutative monoid (= finite multisets) over an observation type `X`. A statistic `f: M(X) → R` is **decomposable** if and only if there exists:

1. A commutative monoid `(S, e, ⊕)`
2. A monoid homomorphism `h: M(X) → S` (i.e., `h(A ⊎ B) = h(A) ⊕ h(B)`, `h(∅) = e`)
3. A finalization function `π: S → R`

such that `f = π ∘ h`.

- If `S = R` and `π = id`, the statistic is **distributive**.
- If `S` is a fixed-size tuple and `π` is not the identity, the statistic is **algebraic**.
- If no finite `S` suffices, the statistic is **holistic**.

## 3. The Fundamental Dichotomy: Batch vs Streaming

### Streaming = Monoid Homomorphism

A streaming aggregation processes observations one at a time (or in chunks) and maintains bounded state. This is exactly a monoid homomorphism:

```
stream: M(X) → S
stream({x₁} ⊎ {x₂} ⊎ ... ⊎ {xₙ}) = h({x₁}) ⊕ h({x₂}) ⊕ ... ⊕ h({xₙ})
```

The observation that streaming algorithms are monoid homomorphisms has been made explicit by multiple authors (Hou 2025, "Stream programs are monoid homomorphisms with state"; Di Lavore & de Felice 2022, "Monoidal Streams for Dataflow Programming").

### The Impossibility Result for Exact Percentiles

**Theorem** (informal): The `p`-th percentile function `Q_p: M(X) → X` for `0 < p < 1` cannot be factored through any finite-dimensional commutative monoid.

**Proof sketch**:

Suppose for contradiction that there exists a commutative monoid `(S, e, ⊕)` with `|S|` bounded, a homomorphism `h: M(X) → S`, and a finalization `π: S → X` such that `Q_p = π ∘ h`.

Consider the multisets `A_k = {k}` for `k = 1, 2, ..., n`. The median of any subset of these singletons depends on *which* elements are present and their count. Specifically, for the median (`p = 0.5`):

- `Q₀.₅({1, 2, 3}) = 2`
- `Q₀.₅({1, 3, 5}) = 3`
- `Q₀.₅({2, 3, 5}) = 3`

For `h` to be a homomorphism: `h({1, 2, 3}) = h({1}) ⊕ h({2}) ⊕ h({3})`. The finalization must recover the median from this. But the number of distinct subsets of `{1, ..., n}` is `2ⁿ`, each potentially having a different median, while `|S|` is fixed. For sufficiently large `n`, the pigeonhole principle guarantees two distinct subsets mapping to the same state but having different medians. Contradiction.

This is why Munro & Paterson's `Ω(n)` single-pass space bound is tight — exact percentiles *require* the full dataset.

### The Approximation Trade-Off

The sketch structures (t-digest, DDSketch, HdrHistogram, Count-Min Sketch) resolve this by relaxing exactness:

1. **DDSketch**: Maps each observation to a logarithmic bucket index `⌊log_γ(x)⌋`. The bucket-count vector forms a commutative monoid under pointwise addition. This *is* an exact monoid homomorphism — the approximation enters only in the finalization (recovering a quantile value from bucket boundaries). Relative error is bounded by `γ - 1` (the bucket spacing parameter).

2. **t-digest**: Maintains a sorted set of centroids (mean, count). Merge is exact (concatenate and re-compress), but the compression step is lossy and order-dependent, so it is only an *approximate* monoid homomorphism. Error is tighter at distribution tails — desirable for latency percentiles (p99, p999).

3. **HdrHistogram**: Uses a fixed set of buckets with sub-exponential spacing (significant-digits precision within each power-of-two range). Merge is exact pointwise addition of bucket counts — a genuine monoid operation. The approximation is in the bucket quantization.

4. **Count-Min Sketch**: A `d × w` matrix of counters. Each observation is hashed by `d` hash functions, incrementing one counter per row. Merge is pointwise addition (exact monoid). Query overestimates frequency by at most `ε·‖a‖₁` with probability `1 - δ`, where `w = ⌈e/ε⌉`, `d = ⌈ln(1/δ)⌉`.

**Key insight**: DDSketch and HdrHistogram are "exact monoid, approximate finalization" — the monoid operation is perfectly associative and commutative; error is introduced only when converting internal state to a human-readable quantile value. t-digest is "approximate monoid" — the compression step breaks strict associativity.

## 4. Constraint Hierarchy on the Observation Type

### Minimum Type Requirements

The type constraints needed for each statistic form a strict hierarchy:

```
Equatable
    └── count (needs nothing beyond element existence)

Comparable (totally ordered)
    ├── min, max
    ├── median, percentiles, rank statistics
    └── mode (needs Hashable, not Comparable — separate branch)

AdditiveArithmetic (has zero, +, -)
    └── sum

AdditiveArithmetic + Comparable + "divisible by natural number"
    ├── mean = sum / count
    └── weighted mean

Embeddable in ℝ (or: multiplication + division + sqrt)
    ├── variance  (requires squaring: x · x)
    ├── standard deviation (requires sqrt)
    ├── covariance, correlation
    └── higher moments (skewness, kurtosis)
```

### Precise Algebraic Requirements

| Statistic | Minimum Structure on T | Why |
|-----------|----------------------|-----|
| count | Any type (even non-Equatable — just count insertions) | Only tracks cardinality |
| min, max | Total order (Comparable) | Need `<` operator |
| sum | Commutative monoid `(T, 0, +)` | Need additive identity and addition |
| mean | Module over Q: `(T, 0, +)` with scalar division by N | `sum / count` requires dividing T by a natural number |
| median, percentiles | Total order (Comparable) | Need to determine rank position |
| variance | `T` must embed into a ring where `T · T` makes sense | Requires `(xᵢ - μ)²`, which squares the type |
| std. deviation | Ring embedding + square root | `sqrt(variance)` must return to original type's units |
| correlation | Field structure (or at least: division of T-products) | `cov(X,Y) / (σ_X · σ_Y)` |

### The "Divisible by Natural" Requirement

Mean requires computing `sum / n` where `n: Int`. This does not require a full field — it requires `T` to be a **module over Q** (or at minimum, divisible by positive integers). For types like `Duration`, this is natural: `(3s + 5s) / 2 = 4s`. For types like `Int`, it requires a decision: integer division (lossy) or promotion to `Double` (changes type).

### In Swift Terms

```swift
// count: no constraint
// min, max: Comparable
// sum: AdditiveArithmetic (provides zero, +, -)
// mean: AdditiveArithmetic + ability to divide by Int
//        (no stdlib protocol for this — custom requirement)
// variance: requires T embeds into a numeric field
//           or: T × T → U where U is the "squared" type
// percentiles: Comparable
```

The gap in Swift's protocol hierarchy: there is no protocol between `AdditiveArithmetic` and `FloatingPoint` that captures "divisible by integer" without requiring full multiplication. This is the `VectorSpace`-over-Q concept that Swift lacks.

## 5. The Unit/Dimension Problem

### How Statistics Transform Units

If observations have type `T` with dimension `[D]`:

| Statistic | Output Dimension | Type Relationship |
|-----------|-----------------|-------------------|
| count | dimensionless (N) | Always `Int` |
| min, max | `[D]` | Same as input: `T → T` |
| sum | `[D]` | Same as input: `T → T` |
| mean | `[D]` | Same as input: `T → T` |
| median, percentiles | `[D]` | Same as input: `T → T` |
| variance | `[D]²` | **Different type**: `T → T²` |
| standard deviation | `[D]` | Same as input: `T → T` (sqrt of variance) |
| covariance(X, Y) | `[D_X] · [D_Y]` | Product of two types |
| correlation | dimensionless | `Double` (always) |
| coefficient of variation | dimensionless | `Double` (always) |

### The Variance Problem

Variance computes `Σ(xᵢ - μ)² / n`. If `xᵢ` has units of seconds, then `(xᵢ - μ)²` has units of seconds². This means:

- `variance(durations)` has type `Duration²` — but `Duration²` is not `Duration`, and Swift has no `Duration.Squared` type.
- `standardDeviation(durations)` has type `Duration` — obtained by `sqrt(variance)`, returning to the original dimension.

This is a genuine type-level problem. Three approaches exist in the literature:

### Approach 1: F# Units of Measure

F# tracks unit exponents at compile time:

```fsharp
[<Measure>] type m  // meters

let variance (xs: float<m> list) : float<m^2> = ...
let stddev (xs: float<m> list) : float<m> = ...
```

The type system naturally represents `m^2` as a distinct type from `m`. Rational exponents are supported. The units are erased at runtime — zero overhead. This is the gold standard for typed dimensional analysis but requires language-level support for unit exponents.

### Approach 2: Boost.Units / mp-units (C++)

C++ template metaprogramming encodes dimensions as compile-time type parameters:

```cpp
quantity<si::length> x = 5.0 * si::meters;
quantity<si::area> var = variance(xs);  // length² = area
quantity<si::length> sd = stddev(xs);   // sqrt(area) = length
```

Dimension arithmetic is resolved at compile time. Like F#, zero runtime cost. The `mp-units` library (Preussner, Pusz; proposed for C++29 as P1935) modernizes this with concepts and stronger type safety.

### Approach 3: Erase Units for Internal Computation

The pragmatic approach used by most systems:

1. Accept observations as `T` (e.g., `Duration`).
2. Project to `Double` for variance computation: `x.asDouble(in: .seconds)`.
3. Variance result is `Double` (caller knows it represents seconds²).
4. Standard deviation result is projected back: `Duration(seconds: sqrt(varianceValue))`.

This loses type safety for variance specifically but avoids the need for a unit-exponent type system.

### Approach 4: Phantom Type Tagging (Available in Swift)

```swift
struct Quantity<Unit: Dimension, Exponent: Exponent> {
    let value: Double
}

// variance returns Quantity<Time, Squared>
// stddev returns Quantity<Time, Identity>
```

This is more limited than F#'s system (no arbitrary rational exponents, no compound units without explicit types) but captures the variance/stddev distinction.

### Recommendation for Swift

For a practical statistics library over typed values (e.g., `Duration`, `Temperature`):

- **min, max, median, mean, percentiles**: Return `T` — no unit problem.
- **variance**: Return `Double` with documentation that the unit is `[T]²`. Alternatively, return an opaque `Variance<T>` wrapper whose only public operation is `squareRoot() → T`.
- **standard deviation**: Return `T` — this is `sqrt(variance)`, which restores the original dimension.
- **correlation, coefficient of variation**: Return `Double` — dimensionless by definition.

The `Variance<T>` wrapper approach is elegant: it makes the squared-unit nature visible in the type system without requiring a general unit-exponent mechanism, and its API surface is minimal (`squareRoot() → T` and perhaps `description`).

## Summary: Design Implications

### For a Streaming/Incremental Statistics API

1. **The monoid is the core abstraction.** Every incrementally computable statistic factors through a commutative monoid homomorphism. The API should expose:
   - `State` (the monoid carrier)
   - `empty` (the identity)
   - `merge(_:_:) → State` (the binary operation)
   - `observe(_:) → State` (the homomorphism from singletons)
   - `finalize(_:) → Result` (the post-processing projection)

2. **Algebraic statistics (mean, variance) use the Chan-Golub-LeVeque monoid** `(count, mean, M2)` as their state. This is a single monoid that supports extracting mean, variance, and standard deviation via different finalization functions.

3. **Holistic statistics require a design choice**: refuse (only support decomposable), require the full dataset (batch-only API), or use a sketch (approximate with bounded error). DDSketch is the strongest candidate for approximate quantiles due to its exact mergeability and relative error guarantee.

4. **Type constraints form a lattice**:
   - `Comparable` enables: count, min, max, percentiles, median.
   - `Comparable + AdditiveArithmetic + "divisible by Int"` enables: + sum, mean.
   - `Comparable + embeds in Double` enables: + variance, stddev, correlation.

   A well-designed API should require only the minimum constraint for each statistic.

5. **Variance is the only common statistic with a unit-dimension mismatch.** The cleanest resolution is an opaque `Variance<T>` type with `squareRoot() → T`.
