## Preamble

This document designs a measurement and statistics type system from first principles. It ignores the existing implementation entirely, building up from nothing to derive the ideal type structure, composition laws, generic stratification, and layering. It then evaluates the design against known anti-patterns and prior art.

**Method**: Build types from the operations they must support, not from the data they hold. Let the algebra dictate the structure.

---

## 1. The Pipeline Model

Every measurement system follows a universal pipeline:

```
Observe  →  Collect  →  Reduce  →  Analyze  →  Present
   T       Sample<T>    Summary     Report      String
```

### Stage Definitions

**Observe**: Produce a single value of type `T` from a system under test.

- The observation is the *raw datum*. It is the type `T` itself — no wrapper needed.
- Metadata (timestamp, context, thread ID) is orthogonal. Attaching it to the observation couples the measurement domain to the observation context. The observer *knows* when and where it observed; the sample does not need to.
- Exception: when the observation *is* a composite (e.g., "Duration + allocation count"), the correct move is a product type at the observation level, not a metadata wrapper. `struct Observation { let elapsed: Duration; let allocated: Int }` — but this is just a custom `T`.

**Conclusion**: `Observation<T>` as a wrapper type is unnecessary. The observation is `T`.

**Collect**: Accumulate observations into a structure.

- The minimal structure is an ordered sequence of `T` values.
- Order matters for two reasons: (1) it preserves temporal information (trend detection, warmup filtering), and (2) percentile computation requires sortable access.
- The structure is `Sample<T>` — a non-empty ordered multiset.
- "Non-empty" is a design choice. An empty sample has no meaningful statistics. But enforcing non-emptiness at the type level (via `init(_ first: T, _ rest: T...)`) is inconvenient for incremental collection. The pragmatic choice: store `[T]`, return optionals or `.zero` for empty samples, document the invariant.

**Reduce**: Transform raw observations into summary statistics.

- A statistic is a function `Sample<T> -> U`.
- For order statistics (min, max, percentile), `U == T` and the constraint is `T: Comparable`.
- For location statistics (mean), `U == T` and the constraint is `T: AdditiveArithmetic` with division by count.
- For dispersion statistics (standard deviation, variance), `U` may differ from `T` — `stddev(Sample<Duration>) -> Duration` but the intermediate computation requires `Double`.
- A `Summary` is a named tuple of computed statistics. It can be a struct or a dictionary, depending on whether the set of statistics is fixed or open.

**Analyze**: Compare, correlate, detect regressions.

- The minimal comparison is `(T_current, T_baseline) -> Change`, where `Change` captures direction and magnitude.
- Regression detection adds a tolerance: `(Change, Double) -> Bool`.
- Correlation relates two samples: `(Sample<T>, Sample<U>) -> Correlation`. This is rarely needed in benchmarking; it matters for multi-metric analysis.

**Present**: Format for human consumption.

- Pure function from analysis results to strings. Should never appear in the same module as the data types.
- Formatting is inherently opinionated (units, precision, color, box-drawing). The data model must be presentation-agnostic.

### Pipeline Independence Principle

Each stage should be implementable without knowledge of the stages before or after it. `Sample<T>` should not know about `ContinuousClock`. `Summary` should not know about ANSI escape codes. `Comparison` should not know about test runners.

## 2. Fundamental Types

### 2.1 `Sample<T: Comparable & Sendable>`

The central type. A sorted collection of observations with derived order statistics.

```swift
public struct Sample<Element: Comparable & Sendable>: Sendable {
    /// The observations in the order they were collected.
    public let values: [Element]

    /// The observations sorted for order-statistic computation.
    /// Computed once at init time, not on every access.
    @usableFromInline
    let sorted: [Element]

    public init(_ values: [Element]) {
        self.values = values
        self.sorted = values.sorted()
    }

    public var count: Int { values.count }
    public var isEmpty: Bool { values.isEmpty }
}
```

**Design decisions**:

1. **Store both ordered and sorted**: Preserving insertion order enables trend analysis (e.g., "are later iterations faster?" — warmup detection). Sorting enables O(1) percentile access after O(n log n) init.

2. **Sort at init, not at access**: Every percentile query on the current implementation calls `durations.sorted()` — O(n log n) per access. Sorting once at init amortizes this cost. The trade-off is doubled memory, which is acceptable for sample sizes typical in benchmarking (10–10,000 elements).

3. **`Comparable & Sendable` as the base constraint**: `Comparable` enables all order statistics. `Sendable` enables safe sharing across concurrency domains (samples are immutable value types, so this is trivially satisfiable for most `T`).

4. **`[Element]` not `NonEmpty<[Element]>`**: Forcing non-emptiness at the type level would require `init(_ first: Element, _ rest: Element...)` or similar, which is awkward for builder patterns and deserialization. Instead, statistical methods return `Optional` or document the precondition.

#### Order Statistics (Tier 0 — `Comparable` only)

```swift
extension Sample {
    public var min: Element? { sorted.first }
    public var max: Element? { sorted.last }

    /// Percentile using the nearest-rank method.
    /// - Parameter p: Percentile in [0, 1].
    /// - Returns: The element at the given percentile, or nil if empty.
    public func percentile(_ p: Double) -> Element? {
        guard !sorted.isEmpty else { return nil }
        let rank = Int((Double(sorted.count) * p).rounded(.down))
        let index = Swift.min(rank, sorted.count - 1)
        return sorted[index]
    }

    public var median: Element? { percentile(0.5) }
    public var p50: Element? { percentile(0.5) }
    public var p75: Element? { percentile(0.75) }
    public var p90: Element? { percentile(0.90) }
    public var p95: Element? { percentile(0.95) }
    public var p99: Element? { percentile(0.99) }
    public var p999: Element? { percentile(0.999) }
}
```

**Why Optional?** A sample with zero elements has no minimum, no maximum, no median. Returning `.zero` (as the current implementation does) silently produces a meaningful-looking but fabricated value. Optionals force the caller to handle the empty case. However, this is a genuine trade-off — pervasive `!` or `?? .zero` at call sites is noisy. An alternative is a `NonEmptySample` wrapper, or precondition-enforced init.

**Percentile method**: The nearest-rank method is the simplest correct algorithm. More sophisticated methods (linear interpolation between ranks, Hyndman-Fan type 7) produce slightly different results for small samples but add complexity. For benchmarking, nearest-rank is standard.

#### Location Statistics (Tier 1 — `AdditiveArithmetic`)

```swift
extension Sample where Element: AdditiveArithmetic {
    /// Sum of all elements.
    public var sum: Element {
        values.reduce(.zero, +)
    }
}
```

Mean requires division by count, which is not expressible with `AdditiveArithmetic` alone. Swift's `Duration` supports `/ Int` but this is not captured by any standard protocol. Options:

**Option A**: Protocol witness

```swift
public protocol Averageable: AdditiveArithmetic {
    static func / (lhs: Self, rhs: Int) -> Self
}

extension Sample where Element: Averageable {
    public var mean: Element {
        guard !values.isEmpty else { return .zero }
        return sum / values.count
    }
}
```

**Option B**: Concrete extensions

```swift
extension Sample where Element == Duration {
    public var mean: Duration { ... }
}
extension Sample where Element == Double {
    public var mean: Double { ... }
}
extension Sample where Element == Int {
    public var mean: Int { ... }
}
```

**Option C**: Conversion-based (Rust criterion approach)

```swift
public protocol ConvertibleToDouble {
    var doubleValue: Double { get }
}

extension Sample where Element: ConvertibleToDouble {
    public var mean: Double {
        guard !values.isEmpty else { return 0 }
        return values.reduce(0.0) { $0 + $1.doubleValue } / Double(values.count)
    }
}
```

**Analysis**: Option A is the cleanest — it introduces a minimal protocol that captures exactly the needed operation. Option B is pragmatic but violates DRY across element types. Option C is lossy (the mean of `Duration` values returns `Double`, not `Duration`).

The ideal design uses Option A, with `Averageable` conformances provided for `Duration`, `Double`, `Int`, `UInt64`.

#### Dispersion Statistics (Tier 2 — requires Double projection)

Standard deviation and variance inherently require floating-point arithmetic (square, square root). Even for `Sample<Duration>`, the computation goes through `Double` internally:

```swift
extension Sample where Element: Averageable & ConvertibleToDouble {
    /// Population standard deviation.
    public var standardDeviation: Double {
        guard count > 1 else { return 0 }
        let m = mean.doubleValue
        let variance = values.reduce(0.0) { acc, x in
            let diff = x.doubleValue - m
            return acc + diff * diff
        } / Double(count - 1)  // Bessel's correction
        return variance.squareRoot()
    }
}
```

For `Sample<Duration>`, a typed extension can return `Duration`:

```swift
extension Sample where Element == Duration {
    /// Standard deviation as a Duration.
    public var standardDeviation: Duration {
        .seconds(standardDeviationAsDouble)
    }

    @usableFromInline
    var standardDeviationAsDouble: Double {
        // ... same computation using .inSeconds
    }
}
```

#### The Constraint Tier Summary

| Tier | Constraint | Operations |
|------|-----------|------------|
| 0 | `Comparable` | min, max, percentile, median, p50–p999 |
| 1 | `Comparable & Averageable` | + sum, mean |
| 2 | Tier 1 + `ConvertibleToDouble` | + standardDeviation, variance, coefficientOfVariation |
| Duration | `Element == Duration` | All of the above with Duration return types |

This is the correct stratification: each tier adds exactly the constraint needed for the operations it unlocks. No over-constraining (requiring `FloatingPoint` for percentiles), no under-constraining (requiring only `Comparable` for mean).

### 2.2 `Accumulator<T>` (Streaming Counterpart)

The O(1) streaming counterpart to `Sample`. A commutative monoid.

```swift
public struct Accumulator: Sendable, Equatable {
    public private(set) var count: UInt64
    public private(set) var sum: UInt64
    public private(set) var min: UInt64
    public private(set) var max: UInt64

    public static let empty = Accumulator(count: 0, sum: 0, min: .max, max: 0)

    public mutating func record(_ value: UInt64) {
        count &+= 1
        sum &+= value
        if value < min { min = value }
        if value > max { max = value }
    }

    /// Commutative, associative merge.
    public func merged(with other: Accumulator) -> Accumulator {
        Accumulator(
            count: count + other.count,
            sum: sum + other.sum,
            min: Swift.min(min, other.min),
            max: Swift.max(max, other.max)
        )
    }

    public var mean: UInt64 {
        guard count > 0 else { return 0 }
        return sum / count
    }
}
```

**Why not generic?** The streaming accumulator operates on `UInt64` because:
1. Nanosecond timestamps are the universal unit for time-based metrics.
2. `UInt64` enables wrapping arithmetic (`&+=`) for zero-cost accumulation under lock.
3. Allocation counts, byte counts, and event counts are all naturally `UInt64`.
4. Genericizing over `T: AdditiveArithmetic & Comparable` adds complexity for no ecosystem benefit — all known use sites operate on integers.

**Why no percentiles?** Percentiles require either (a) storing all values (Sample) or (b) an approximate data structure (HDR histogram, t-digest, quantile sketch). Option (a) defeats the purpose of streaming. Option (b) is a significant increase in complexity. The Accumulator captures the statistics that are decomposable (count, sum, min, max). Percentile approximation belongs in a separate `Histogram` type.

### 2.3 The `Metric` Selector

A `Metric` is a first-class function `Sample<T> -> T`:

```swift
public enum Metric: String, Sendable, Codable, CaseIterable {
    case min
    case max
    case median
    case mean
    case p50
    case p75
    case p90
    case p95
    case p99
    case p999
}
```

But `extract(from:)` has a problem: it needs different constraints depending on the case. `.min` needs `Comparable`; `.mean` needs `Averageable`. Options:

**Option A**: Restrict the enum to `Sample where Element: Averageable`

```swift
extension Metric {
    public func extract<T: Averageable & Comparable & Sendable>(from sample: Sample<T>) -> T? {
        switch self {
        case .min: return sample.min
        case .max: return sample.max
        case .median: return sample.median
        case .mean: return sample.mean
        case .p50: return sample.p50
        case .p75: return sample.p75
        case .p90: return sample.p90
        case .p95: return sample.p95
        case .p99: return sample.p99
        case .p999: return sample.p999
        }
    }
}
```

**Option B**: Two enums — `OrderMetric` (Comparable-only) and `LocationMetric` (+ mean)

**Analysis**: Option A is pragmatic. In practice, you never use `Metric` with a type that does not support mean — the whole point is to select among available statistics. If `T` does not support mean and you select `.mean`, that is a logical error that should not compile. Option A catches it at the constraint level.

### 2.4 `Comparison` — Relating Two Measurements

```swift
public struct Comparison<T: Comparable & Sendable>: Sendable {
    public let baseline: Sample<T>
    public let current: Sample<T>
    public let metric: Metric

    /// Direction: lower-is-better (latency) or higher-is-better (throughput).
    public let polarity: Polarity

    public enum Polarity: Sendable {
        case lowerIsBetter   // latency, memory, allocations
        case higherIsBetter  // throughput, ops/sec
    }
}
```

The critical insight: **polarity determines whether a change is a regression or improvement**. The current implementation hardcodes `isRegression = change > 0`, which assumes lower-is-better. This silently produces wrong results for throughput benchmarks.

```swift
extension Comparison where T: Averageable & ConvertibleToDouble {
    /// Percentage change: positive means the measured quantity increased.
    public var change: Double {
        guard let cur = metric.extract(from: current),
              let base = metric.extract(from: baseline) else { return 0 }
        let c = cur.doubleValue
        let b = base.doubleValue
        guard b != 0 else { return 0 }
        return (c - b) / b
    }

    /// Whether the change represents a regression (performance got worse).
    public var isRegression: Bool {
        switch polarity {
        case .lowerIsBetter: return change > 0   // value went up = bad
        case .higherIsBetter: return change < 0   // value went down = bad
        }
    }

    /// Whether the change represents an improvement.
    public var isImprovement: Bool {
        switch polarity {
        case .lowerIsBetter: return change < 0
        case .higherIsBetter: return change > 0
        }
    }

    /// Whether the change exceeds a tolerance threshold.
    public func exceedsTolerance(_ tolerance: Double) -> Bool {
        abs(change) > tolerance && isRegression
    }
}
```

### 2.5 `Summary` — Named Collection of Statistics

A `Summary` captures the result of reducing a sample to a fixed set of statistics:

```swift
public struct Summary<T: Sendable>: Sendable {
    public let count: Int
    public let min: T
    public let max: T
    public let median: T
    public let mean: T
    public let p95: T
    public let p99: T
    public let standardDeviation: T?  // nil if T does not support it
}
```

**Is Summary generic or concrete?** Generic `Summary<T>` allows `Summary<Duration>`, `Summary<Int>`, `Summary<Double>`. But `standardDeviation` being optional (only available when `T: ConvertibleToDouble`) is awkward. Alternative: two summary types, `OrderSummary<T>` (count/min/max/median/percentiles) and `FullSummary<T>` (adds mean/stddev). This follows the constraint tiering.

However, `Summary` is primarily a presentation type — it exists to be formatted into a report. Keeping it as a simple struct with all fields (some possibly nil) is more practical than a two-type hierarchy.

## 3. Composition Laws

### 3.1 Sample + Sample -> Sample (Concatenation)

```swift
extension Sample {
    /// Merge two samples by concatenating their observations.
    /// Preserves insertion order within each sample; `self` values come first.
    public func merged(with other: Sample<Element>) -> Sample<Element> {
        Sample(values + other.values)
    }
}
```

Properties:
- Associative: `(a.merged(b)).merged(c) == a.merged(b.merged(c))`
- Not commutative (order depends on which comes first, though statistics are order-independent)
- Identity: `Sample([])` (empty sample)
- This makes `(Sample<T>, merged, Sample([]))` a monoid

### 3.2 Accumulator + Accumulator -> Accumulator (Merge)

```swift
a.merged(with: b) == b.merged(with: a)  // commutative
```

Properties:
- Commutative and associative
- Identity: `Accumulator.empty`
- This makes `(Accumulator, merged, .empty)` a *commutative monoid*

The commutativity matters: accumulators from different threads can be merged in any order. This enables lock-free per-thread accumulators with periodic merge.

### 3.3 Sample -> Accumulator (Homomorphism)

```swift
extension Accumulator {
    /// Construct from a sample by folding all values.
    public static func from<T: ConvertibleToUInt64>(_ sample: Sample<T>) -> Accumulator {
        var acc = Accumulator.empty
        for value in sample.values {
            acc.record(value.uint64Value)
        }
        return acc
    }
}
```

This is a lossy homomorphism: `from(a.merged(b)) == from(a).merged(from(b))`, but `from` is not injective (many samples produce the same accumulator).

### 3.4 Accumulator -/-> Sample (No inverse)

An accumulator with `{ count: 3, sum: 6, min: 1, max: 3 }` could have come from `[1, 2, 3]` or `[1, 3, 2]` at minimum. The full observation sequence is irrecoverable. Percentiles cannot be computed from an accumulator.

### 3.5 Category-Theoretic View

```
Sample<T> ----fold----> Accumulator
    |                       |
    | statistics            | mean/min/max
    v                       v
 Summary<T>             (UInt64, UInt64, UInt64)
```

`Sample` is the free object — it retains all information. `Accumulator` is the quotient — it collapses the sample down to decomposable aggregates. The `fold` arrow is the canonical surjection. Any statistic computable from an `Accumulator` is also computable from a `Sample` (the homomorphism preserves those statistics). But not vice versa.

## 4. The Generic vs. Specialized Trade-off

### 4.1 The Problem

A fully generic `Sample<T: Comparable>` provides percentiles for any ordered type. But:

- `mean` needs `T + T -> T` and `T / Int -> T` (not in any standard Swift protocol)
- `stddev` needs `T -> Double` (not in any standard Swift protocol)
- `Duration` supports `+` and `/ Int` but does not conform to `AdditiveArithmetic` in the stdlib (it does, actually, since SE-0329)
- `Int` supports `+` and `/` but integer division truncates

### 4.2 How Other Languages Handle This

**Haskell**: Typeclasses with hierarchical constraints.
```haskell
class Ord a => OrderStatistics a where
    percentile :: Double -> Sample a -> a
class (OrderStatistics a, Fractional a) => LocationStatistics a where
    mean :: Sample a -> a
```
Haskell can express this cleanly because typeclasses compose and constraints propagate.

**Rust criterion**: The `Measurement` trait requires `to_f64()`. Everything operates on `f64` internally. This is simple but lossy — you cannot recover the original type from the statistical output.

```rust
pub trait Measurement {
    type Value;
    fn to_f64(&self, value: &Self::Value) -> f64;
}
```

**Scala**: Implicit evidence parameters.
```scala
def mean[T](sample: Sample[T])(implicit ev: Numeric[T]): T
```

**BenchmarkDotNet (.NET)**: Takes `double[]`. All input is converted to double before any statistics. Simple. Lossy.

### 4.3 The Swift Answer: Conditional Conformance + Protocol Witnesses

Swift's conditional conformance is exactly the right tool:

```swift
// Base type — Comparable only
public struct Sample<Element: Comparable & Sendable>: Sendable { ... }

// Tier 0: order statistics (on the base type)
extension Sample {
    public var min: Element? { ... }
    public func percentile(_ p: Double) -> Element? { ... }
}

// Tier 1: location statistics (conditional extension)
extension Sample where Element: Averageable {
    public var mean: Element { ... }
}

// Tier 2: dispersion statistics (conditional extension)
extension Sample where Element: Averageable, Element: ConvertibleToDouble {
    public var standardDeviation: Double { ... }
    public var variance: Double { ... }
}

// Tier D: Duration-specific (concrete type extension)
extension Sample where Element == Duration {
    public var standardDeviation: Duration { ... }
}
```

This is strictly better than the alternatives because:
1. **No information loss**: `mean` of `Sample<Duration>` returns `Duration`, not `Double`.
2. **No over-constraining**: you can compute percentiles on any `Comparable` type without needing arithmetic.
3. **Compile-time safety**: calling `.mean` on `Sample<SomeComparableButNotAverageable>` is a compile error.
4. **Retroactive conformance**: users can make their own types `Averageable` to opt into mean computation.

### 4.4 The "Always Convert to Double" Alternative

Rust criterion's approach: define a `to_f64()` method and operate entirely in `Double` space.

Advantages:
- Simpler type signatures
- One implementation for all statistics
- No protocol proliferation

Disadvantages:
- Lossy: `Duration` -> `Double` -> `Duration` introduces floating-point rounding
- Type information lost: the mean of `Sample<Duration>` should be `Duration`, not `Double`
- Unnecessary for order statistics: percentiles on Duration do not need Double conversion

**Verdict**: The tiered conditional conformance approach is superior for Swift. The "always convert" approach is a concession to languages without conditional conformance.

## 5. Ideal Call Sites

### 5.1 Benchmarking

```swift
// Collect observations
let sample: Sample<Duration> = measure(iterations: 1000) { doWork() }

// Order statistics (Tier 0)
let fastest = sample.min       // Duration?
let p99 = sample.p99           // Duration?
let p99th = sample.percentile(0.99)  // Duration?

// Location statistics (Tier 1)
let avg = sample.mean           // Duration (not Optional — non-empty after measure())

// Dispersion statistics (Tier 2)
let spread = sample.standardDeviation  // Duration

// Metric selector
let value = Metric.p95.extract(from: sample)  // Duration?

// Summary (all stats at once)
let summary = sample.summary    // Summary<Duration>
```

### 5.2 Comparison / Regression Detection

```swift
let baseline: Sample<Duration> = loadBaseline("sort-benchmark")
let current: Sample<Duration> = measure(iterations: 1000) { sort(data) }

let comparison = Comparison(
    baseline: baseline,
    current: current,
    metric: .p99,
    polarity: .lowerIsBetter
)

comparison.change          // Double (e.g., 0.15 = 15% increase)
comparison.isRegression    // Bool
comparison.exceedsTolerance(0.10)  // Bool (true if >10% regression)

// Assertion
try expectNoRegression(
    current: current,
    baseline: baseline,
    metric: .median,
    tolerance: 0.10,
    polarity: .lowerIsBetter
)
```

### 5.3 Streaming Telemetry

```swift
var acc = Accumulator.empty

// Hot path — O(1) per record
acc.record(elapsedNs)
acc.record(elapsedNs)

// Snapshot for reporting
let snapshot = acc  // value type, copy is the snapshot
snapshot.count   // 2
snapshot.mean    // UInt64

// Merge from multiple threads
let merged = thread1Acc.merged(with: thread2Acc)
```

### 5.4 Generic (Non-Duration)

```swift
// Allocation counts
let allocSample = Sample<Int>([1024, 2048, 1024, 4096, 2048])
let p95 = allocSample.p95       // Int?
let avgAlloc = allocSample.mean // Int (via Averageable conformance on Int)

// Throughput
let opsSample = Sample<Double>([1_000_000, 1_100_000, 950_000])
let medianOps = opsSample.median  // Double?
let meanOps = opsSample.mean      // Double
```

### 5.5 Builder Pattern for Incremental Collection

```swift
var builder = Sample<Duration>.Builder()
for _ in 0..<100 {
    let start = ContinuousClock.now
    doWork()
    builder.append(ContinuousClock.now - start)
}
let sample = builder.build()  // Sample<Duration>
```

## 6. The Platonic Layering

### Proposed Four-Layer Architecture

```
Layer 3: Test Integration
         - Test runner trait integration (.timed())
         - Regression assertions (expectNoRegression)
         - Console reporting (printPerformance, box drawing)
         - Suite orchestration
         - CI tooling, baseline storage

Layer 2: Benchmark Engine
         - measure() function (ContinuousClock integration)
         - Warmup logic
         - Multi-iteration collection
         - Comparison.formatted()

Layer 1: Statistical Data Model
         - Sample<T>
         - Metric enum
         - Comparison<T>
         - Summary<T>
         - Accumulator
         - Averageable protocol

Layer 0: (not needed)
```

### Why Not Layer 0 (Algebraic Abstractions)?

A `Monoid` protocol (`associatedtype Element; static var identity: Element; func combine(Element, Element) -> Element`) would let us express `Sample.merged` and `Accumulator.merged` as monoid instances. But:

1. Swift does not have a standard `Monoid` protocol, and introducing one creates ecosystem fragmentation.
2. The concrete merge methods are self-documenting; the abstraction adds indirection without enabling new call sites.
3. Haskell's `statistics` package does *not* abstract over monoid structure — it just uses `Semigroup` and `Monoid` from `base`. Swift has no equivalent of `base`.

**Verdict**: Layer 0 algebraic abstractions are not worth the cost. Document the monoidal structure in comments; do not encode it in types.

### Mapping to Swift Institute Five-Layer Architecture

| Pipeline Layer | Swift Institute Layer | Package |
|---------------|----------------------|---------|
| Statistical Data Model | **Layer 1 (Primitives)** | `swift-sample-primitives` |
| Benchmark Engine | **Layer 3 (Foundations)** | `swift-tests` (Tests Benchmark) |
| Test Integration | **Layer 3 (Foundations)** | `swift-tests` (Tests Performance) |
| Reporting | **Layer 3 (Foundations)** | `swift-tests` (same module) |

The statistical data model belongs at Layer 1 because:
- Zero external dependencies (pure `Comparable` computation)
- Reusable by any layer (IO metrics, pool metrics, PDF rendering benchmarks)
- Domain-independent (works for Duration, Int, Double, UInt64, any Comparable)

The benchmark engine and test integration belong at Layer 3 because:
- They depend on `ContinuousClock` (stdlib, but conceptually a system resource)
- They depend on `Console` for formatting
- They depend on test framework concepts (traits, runners)

### Alternative: Three-Package Split (Haskell Model)

```
swift-sample-primitives     — Sample<T>, Metric, Accumulator
swift-benchmark             — measure(), warmup, iteration
swift-tests                 — test integration, reporting, assertions
```

This matches Haskell's `statistics` / `criterion-measurement` / `criterion` split. The benefit: `swift-benchmark` can be used by applications (Layer 5) without pulling in test infrastructure. The cost: one more package to manage.

**Recommendation**: Start with the two-package split (primitives + swift-tests). If application-layer benchmarking emerges as a need, extract `swift-benchmark` then.

## 7. Anti-Patterns to Avoid

### 7.1 Coupling Data Collection to Analysis (Go `testing.B`)

Go's `testing.B` struct is both the runner and the result container. You call `b.N` to get iteration count and `b.ReportMetric` to record values, but you cannot extract the raw timings for offline analysis. The benchmark result is inseparable from the benchmark execution.

**How to avoid**: `measure()` returns `Sample<Duration>`. The caller decides what to do with it — compute statistics, compare to baseline, serialize, ignore. The measurement function owns collection; the caller owns analysis.

### 7.2 Coupling Analysis to Presentation (JMH)

JMH's `Result` hierarchy bundles statistical computation with formatting. `RunResult` contains both the computed statistics and the format strings. You cannot get the numbers without the presentation overhead.

**How to avoid**: `Sample<T>` and `Summary<T>` are presentation-free value types. Formatting lives in a separate module (or at minimum a separate extension file) that imports `Console`.

### 7.3 Over-Computation

Computing all statistics when you only need one is wasteful. The current implementation's `printPerformance` computes min, median, mean, p95, p99, max, and stddev — seven statistics — even when you only need median for a threshold check.

**How to avoid**: Statistics are lazy properties computed on demand, not eagerly computed at init. `sample.p99` computes only p99. `sample.summary` computes everything (opt-in). The `Metric` enum + `extract(from:)` computes exactly one statistic.

Caveat: if `sorted` is computed at init (as proposed above), the O(n log n) cost is paid regardless. But the O(1) per-statistic cost after sorting is negligible. The real savings come from not computing stddev (which requires a full-sample iteration) when you do not need it.

### 7.4 Premature Double Conversion

Converting to `Double` too early loses type information. If `measure()` returns `Double` instead of `Duration`, the caller cannot format the result with appropriate units, convert to different time scales, or compare against `Duration` thresholds.

**How to avoid**: Generic `Sample<T>` preserves `T` throughout. Conversion to `Double` happens only for statistics that require it (stddev), and even then, the result can be converted back to `T` (e.g., `Duration.seconds(stddevAsDouble)`).

### 7.5 DRY Violation Across Measurement Contexts

The current codebase has `Tests.Measurement` and `Test.Benchmark.Measurement` — two nearly identical types with identical statistical methods, in two different modules, each with their own `Metric` enum and `printPerformance` function.

**How to avoid**: One `Sample<T>` type at Layer 1. Both `Tests Performance` and `Tests Benchmark` depend on it. The duplication is structurally impossible because there is only one source of truth.

### 7.6 Hardcoded Polarity

Assuming "lower is better" works for latency but fails for throughput. The current `Comparison.isRegression` computes `change > 0` without considering whether the metric measures something you want to minimize or maximize.

**How to avoid**: `Comparison` takes a `Polarity` parameter (`.lowerIsBetter` or `.higherIsBetter`) that determines regression direction. Default to `.lowerIsBetter` for backward compatibility.

### 7.7 Empty-Sample Silent Zeroing

Returning `.zero` for empty samples hides bugs. If someone accidentally creates an empty `Measurement`, all statistics report zero — which looks like an impossibly fast benchmark result rather than a missing-data error.

**How to avoid**: Return `Optional` from statistical methods, or enforce non-emptiness at `init`. The `measure()` function always produces non-empty samples (it runs at least one iteration), so call sites after `measure()` can safely force-unwrap if `count` is checked.

## 8. Open Questions

### 8.1 Sorted Storage: Always or On-Demand?

Sorting at init doubles memory but enables O(1) percentile access. For small samples (typical in benchmarking: 10–1000 elements), this is negligible. For large samples (production telemetry with 1M elements), it may matter. Options:

- **Always sort at init** (proposed above): simple, fast access, 2x memory
- **Sort on first access and cache**: lazy, but thread-safety complexity for value types
- **Sort on every access**: no extra memory, O(n log n) per access (current behavior)
- **Provide both**: `Sample` (always sorted) and `LazySample` (sort on demand)

Recommendation: Always sort at init. Benchmarking samples are small. If production telemetry needs millions of elements, it should use `Accumulator` or `Histogram`, not `Sample`.

### 8.2 NonEmpty Enforcement

Should `Sample` enforce non-emptiness at the type level?

- **Yes**: Eliminates the "zero for empty" anti-pattern. All statistics return non-optional.
- **No**: Enables builder patterns, deserialization, and test fixtures with empty samples.

Recommendation: Do not enforce at the type level. Instead, provide a checked constructor:

```swift
extension Sample {
    public init?(nonEmpty values: [Element]) {
        guard !values.isEmpty else { return nil }
        self.init(values)
    }
}
```

And document that `measure()` always returns non-empty samples.

### 8.3 Interpolation in Percentile Computation

The nearest-rank method (used here and in the current implementation) returns the element at rank `floor(count * p)`. The linear interpolation method (Excel's `PERCENTILE.INC`, numpy default, Hyndman-Fan Type 7) interpolates between adjacent ranks. For small samples, the difference can be significant.

Recommendation: Use nearest-rank for `Comparable`-only (interpolation requires arithmetic). Provide an `interpolatedPercentile` method for `Averageable` elements.

### 8.4 Accumulator Genericity

Should `Accumulator` be generic over `UInt64` vs `Int64` vs `Duration`? The current IO implementation uses `UInt64` nanoseconds. A generic accumulator would need `Comparable & AdditiveArithmetic` and wrapping arithmetic support.

Recommendation: Keep `Accumulator` concrete on `UInt64`. It is an optimization type for hot paths. Genericity adds overhead (no wrapping arithmetic on generic `AdditiveArithmetic`). Callers convert at the boundary.

### 8.5 Protocol Naming

`Averageable` is the best name I can find for "supports addition and division by integer count." Alternatives considered:

- `Divisible` — too broad (implies arbitrary division)
- `ScalableByInt` — awkward
- `HasMean` — couples the protocol to one use
- `VectorSpace` — mathematically correct but pretentious and over-abstract for Swift
- `MeanComputable` — compound name, violates [API-NAME-002]

`Averageable` is not ideal but is the least bad option. It clearly communicates the intent: "types for which an average can be computed."

## 10. Summary of Recommendations

| Decision | Recommendation | Rationale |
|----------|---------------|-----------|
| Observation wrapper type | No — raw `T` is sufficient | Metadata is orthogonal; coupling it to observation adds noise |
| Sample storage | Store both insertion-order and sorted | Enables both trend analysis and O(1) percentile access |
| Empty sample handling | Return Optional from statistics | Prevents silent zero-value bugs |
| Generic stratification | Three constraint tiers via conditional extension | Matches the mathematical requirements exactly |
| Mean computation | `Averageable` protocol (`AdditiveArithmetic` + `/ Int`) | Minimal protocol that captures exactly the needed operation |
| Stddev computation | Requires `ConvertibleToDouble`; Duration extension returns Duration | Floating-point computation is unavoidable; typed return preserves usability |
| Accumulator genericity | Concrete on `UInt64` | Hot-path optimization type; genericity adds overhead for no ecosystem benefit |
| Metric type | Enum with `extract(from:)` constrained to `Averageable` | First-class function from sample to scalar |
| Comparison polarity | Explicit `.lowerIsBetter` / `.higherIsBetter` parameter | Prevents silent wrong-direction regression for throughput metrics |
| Layering | Sample + Metric + Accumulator at L1; measure + assertions + reporting at L3 | Data model is domain-independent; runner is domain-specific |
| Percentile method | Nearest-rank (default), interpolated (opt-in for Averageable) | Nearest-rank needs only Comparable; interpolation needs arithmetic |
| Sorted storage timing | At init, not on access | Sample sizes are small in benchmarking; amortized cost is negligible |
