## 2. Observability Classification

| Category | Types | Notes |
|----------|-------|-------|
| **Counter** (monotonically increasing) | `Pool.Metrics.created/fills/closed/acquisitions/releases/timeouts`, `Memory.Allocation.Statistics.allocations/deallocations` | Scalar, never decreasing. No statistical aggregation needed. |
| **Gauge** (current value) | `Pool.Metrics.checkedOut/available/waiters/peakCheckedOut`, `Memory.Allocation.Statistics.bytesAllocated` | Point-in-time snapshot. No distribution. |
| **Histogram/Summary** (distribution of samples) | `Tests.Measurement`, `Test.Benchmark.Measurement` | Ordered collection with percentile/mean/stddev. **This is the only category that benefits from generalization.** |

Key observation: Counters and gauges are structurally different from distributions. They are single scalars, not collections. A generic `Sample<T>` would only apply to the histogram/summary category.

## 3. Statistical Computation Analysis

### Operation-by-operation type requirements

| Operation | Current impl | Requires from `T` | Duration-specific? |
|-----------|-------------|-------------------|-------------------|
| `min` | `durations.min()` | `Comparable` | No |
| `max` | `durations.max()` | `Comparable` | No |
| `percentile(_ p:)` | sort + index | `Comparable` | No |
| `median` | `percentile(0.5)` | `Comparable` | No |
| `p50/p75/p90/p95/p99/p999` | `percentile(n)` | `Comparable` | No |
| `mean` | `reduce(.zero, +) / count` | `AdditiveArithmetic` + division by `Int` | **Partially** -- needs `.zero` and `+` (from `AdditiveArithmetic`) plus `/ Int` |
| `standardDeviation` | convert to `Double`, compute variance, convert back | **Yes** -- uses `.inSeconds` and `.seconds(_:)` | **Yes** |

### Generalization feasibility per operation

**Fully generic** (only `Comparable`): `min`, `max`, `percentile`, `median`, all `pNN` accessors. These would work on `[Int]`, `[Double]`, `[Duration]`, `[UInt64]` -- anything `Comparable`.

**Generic with constraints** (`AdditiveArithmetic` + division): `mean` requires `reduce(.zero, +)` which needs `AdditiveArithmetic`, plus division by `Int`. For `Duration`, `/` is stdlib-provided. For `Int` and `Double`, `/` is available. A protocol witness for "dividable by Int" is needed -- this maps naturally to a hypothetical constraint or to requiring `BinaryInteger` or `FloatingPoint`.

**Not directly generic**: `standardDeviation` converts to `Double` via `.inSeconds`, computes in floating-point space, and converts back via `.seconds(_:)`. This requires a bidirectional `Double` conversion. Two approaches:
  1. A protocol like `protocol Measurable { var toDouble: Double { get }; static func fromDouble(_ d: Double) -> Self }` -- this is ad-hoc and ugly.
  2. Compute stddev only for types that are themselves `FloatingPoint`, and provide `Duration`-specific overloads. This is more honest.
  3. Accept that stddev returns `Double` always (the variance is dimensionless after normalization, but the stddev has the same units as the original, so returning `Double` loses unit information).

### Could `percentile(_ p:)` work on `[Int]` or `[Double]`?

Yes, trivially. The implementation is:
```swift
let sorted = values.sorted()
let index = Int(Double(sorted.count) * p)
let clampedIndex = Swift.min(index, sorted.count - 1)
return sorted[clampedIndex]
```
This requires only `Comparable` (for `sorted()`). It works on any `Comparable` collection element.

### Minimum constraint sets

| Tier | Constraint | Enables |
|------|-----------|---------|
| 1 | `Comparable` | min, max, percentile, median, all pNN |
| 2 | `Comparable & AdditiveArithmetic` + `/ Int` | + mean |
| 3 | Tier 2 + `Double` conversion | + standardDeviation |

## 4. Comparison Type Analysis

### Does regression detection require Duration?

The current `Tests.Comparison` implementation:

```swift
public var change: Double {
    let cur = currentValue.inSeconds    // Duration -> Double
    let base = baselineValue.inSeconds  // Duration -> Double
    return (cur - base) / base          // pure Double arithmetic
}
```

The `.inSeconds` call is the only Duration-specific part. The actual regression math is `(current - baseline) / baseline`, which is a generic percentage-change calculation.

For `Int` (allocation counts): `change = Double(current - baseline) / Double(baseline)` -- trivially generic.
For `Double` (throughput): `change = (current - baseline) / baseline` -- already generic.

**Conclusion**: Regression detection does NOT require `Duration`. It requires converting `T` to `Double` for the ratio calculation. This is a weaker requirement than stddev -- it only needs a one-way conversion to `Double`.

The `isRegression`/`isImprovement` semantics (positive change = regression) assume "lower is better" (latency). For throughput metrics, "higher is better" and the polarity inverts. A generic comparison type would need a `direction` parameter (`.lowerIsBetter` / `.higherIsBetter`).

## 5. Naming Analysis

### Candidates

| Name | Meaning | Precedent |
|------|---------|-----------|
| `Measurement` | Act of measuring; a single measured value or collection | Apple's `Foundation.Measurement<UnitType>` (but that's a single scalar with units, not a distribution) |
| `Sample` | A subset of observations drawn from a population | Statistics: correct term. But also means "a single data point" in common usage (ambiguous). |
| `Distribution` | The shape of values across a range | Statistics: describes the abstract shape, not the concrete data. Implies continuous PDF, not a bag of observations. |
| `Summary` | Aggregated statistics from a set of observations | Prometheus: `summary` is a client-side quantile computation. Close match. |
| `Histogram` | Bucketed frequency distribution | Prometheus/OpenTelemetry: pre-defined bucket boundaries with counts. Our type has no buckets -- it stores raw values. Not a match. |
| `Digest` | Compressed statistical summary | t-digest, HdrHistogram. Implies lossy compression. Our type stores all raw values. Not a match. |

### Observability standard terminology

**OpenTelemetry** defines three metric instruments:
- **Counter**: monotonically increasing sum
- **Gauge**: current value
- **Histogram**: distribution of values into configured buckets

OpenTelemetry's `Histogram` records individual measurements and aggregates them into bucket counts + sum + count + min + max. It does NOT compute percentiles client-side (that's done by the backend from bucket boundaries). Our type computes percentiles from raw values, which is closer to a **Summary**.

**Prometheus** defines:
- **Summary**: client-side streaming quantile computation (phi-quantiles). Stores quantile values, sum, count.
- **Histogram**: server-side bucketed distribution.

Our type is closest to a Prometheus **Summary** -- it stores raw observations and computes quantiles directly.

### Recommendation

**`Sample`** is the best name, despite the single-vs-plural ambiguity, for these reasons:

1. **Statistical correctness**: In statistics, a "sample" is a collection of observations from a population. `Sample<Duration>` reads as "a sample of Duration observations" -- exactly what this is.

2. **Avoids Foundation collision**: `Measurement` collides with `Foundation.Measurement<UnitType>`, which is a single scalar with dimensional analysis. Using `Measurement` for a distribution is confusing.

3. **Generic-friendly**: `Sample<Int>` (allocation counts), `Sample<Double>` (throughput), `Sample<Duration>` (latency) all read naturally.

4. **Not `Summary`**: While Prometheus calls this concept a "summary", that name implies pre-computed aggregates. Our type stores raw values and computes statistics on demand. `Sample` better describes the raw data.

5. **Not `Distribution`**: A distribution is the mathematical abstraction (PDF/CDF). Our type is the empirical data, not the abstraction.

6. **Nest.Name compliance**: `Tests.Sample<Duration>` or standalone `Sample<Duration>` both satisfy [API-NAME-001]. If placed in a statistics namespace: `Statistics.Sample<Duration>`.

### Alternative: keep `Measurement` for the Duration-specific type

If the generic version is introduced alongside the existing type, the naming could be:
- `Sample<T>` -- generic, provides Tier 1 (Comparable) operations
- `Tests.Measurement` -- Duration-specific, wraps `Sample<Duration>` and adds stddev + `.inSeconds`-dependent operations

This layering avoids forcing a `toDouble` protocol onto the generic type.

---

## Summary of Findings

1. **Two identical implementations exist** (`Tests.Measurement` and `Test.Benchmark.Measurement`). This is the strongest argument for extraction -- even without generalization, deduplication is needed.

2. **80% of statistical operations are generic over `Comparable`** (min, max, percentile, median, all pNN). Only mean requires `AdditiveArithmetic`, and only stddev requires Duration-specific conversion.

3. **A tiered approach works**: Extract `Sample<T: Comparable>` with percentile/min/max, conditionally extend for `mean` where `T: AdditiveArithmetic`, and provide Duration-specific extensions for stddev.

4. **`Comparison` can be generalized** with a `toDouble` conversion and a direction parameter, but this adds complexity. The simpler path: keep `Comparison` Duration-specific for now, and generalize only when a concrete non-Duration comparison use case appears.

5. **Recommended name**: `Sample<T>`. Statistically accurate, generic-friendly, avoids Foundation collision, compliant with naming rules.

6. **Layer placement question**: If `Sample` is purely `Comparable`-constrained with no dependencies, it could live in primitives. If it needs `Time_Primitives` for Duration extensions, it stays in foundations. The generic core (`Sample<T: Comparable>`) is a primitives-tier concept; the Duration-specific extensions belong in foundations.
