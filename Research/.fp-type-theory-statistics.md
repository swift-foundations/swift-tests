## 1. Haskell's `statistics` Package (Bryan O'Sullivan)

### Type Hierarchy

The package defines a clean separation between data, estimation, and error modeling:

```haskell
-- Data layer: monomorphic
type Sample         = Vector Double
type WeightedSample = Vector (Double, Double)

-- Estimation layer: parameterized over error model
data Estimate e a = Estimate
  { estPoint :: !a        -- point estimate
  , estError :: !(e a)    -- error representation (polymorphic)
  }

-- Error models
newtype NormalErr a = NormalErr { normalError :: a }

data ConfInt a = ConfInt
  { confIntLDX :: !a           -- lower deviation
  , confIntUDX :: !a           -- upper deviation
  , confIntCL  :: !(CL Double) -- confidence level
  }

-- Confidence level (stores 1-CL internally for precision near 1)
newtype CL a = ...
newtype PValue a = ...
```

### Why Monomorphic Double for Sample?

`Sample = Vector Double` is a deliberate choice, not an oversight. The reasoning:

1. **SIMD vectorization**: Unboxed `Vector Double` maps directly to contiguous memory, enabling hardware-level parallelism. Generic vectors lose this.
2. **Numerical stability**: All algorithms (Welford's mean, two-pass variance, bootstrap) are tuned for IEEE 754 Double. Generics would require ad-hoc numeric type classes with unpredictable precision.
3. **API simplicity**: Every function signature becomes `Vector v Double => v Double -> Double` rather than carrying `(Floating a, Ord a, Unbox a, ...)` constraint stacks.
4. **Honest acknowledgment**: Heavy statistics *will* use Double. Pretending otherwise adds type machinery with no numerical benefit.

### The Estimate Parameterization

`Estimate e a` parameterizes over the error model `e`. This is the key insight:

- `Estimate NormalErr Double` -- symmetric Gaussian error
- `Estimate ConfInt Double` -- asymmetric bootstrap confidence interval
- `Estimate UpperLimit Double` -- one-sided bound

The point estimate is always `a`, but different analysis methods produce different error characterizations. By making `e` a type parameter, the same `Estimate` type serves both parametric and non-parametric inference without runtime dispatch.

### Resampling Composition

```haskell
newtype Resample = Resample { fromResample :: Vector Double }

resample :: GenIO -> [Estimator] -> Int -> Vector Double
         -> IO [(Estimator, Bootstrap Vector Double)]
```

The composition path: `Sample` (raw data) feeds into `resample`, which produces `Bootstrap Vector Double` (the resampled distributions), which feeds into `bootstrapBCA` or `basicBootstrap`, which produces `Estimate ConfInt Double`. The `Resample` newtype exists solely to prevent confusion between original samples and resampled data -- a type-level distinction that catches bugs at compile time.

### RNG Separation (mwc-random)

`mwc-random` is a separate package because:

1. **Dependency minimization**: Not every consumer of `statistics` needs RNG. Separating it lets pure analysis code avoid IO dependencies.
2. **Pluggability**: Issue #164 on haskell/statistics discusses supporting alternative RNGs. A hard-wired dependency prevents this.
3. **Performance isolation**: `mwc-random` is tuned for speed (2-3x faster than Mersenne Twister) with all operations in IO/ST. This is a fundamentally different optimization target than statistical computation.
4. **Layering principle**: RNG is infrastructure; statistics is analysis. Coupling them violates separation of concerns.

## 2. Haskell's `criterion-measurement`

### Exported Types

The package exports two key types:

```haskell
data Measured = Measured
  { measTime                :: !Double    -- wall-clock seconds
  , measCpuTime             :: !Double    -- CPU seconds (user + kernel)
  , measCycles              :: !Int64     -- rdtsc cycles
  , measIters               :: !Int64     -- loop iterations
  -- GC fields (only available with +RTS -T)
  , measAllocated           :: !Int64     -- bytes allocated
  , measNumGcs              :: !Int64     -- GC count
  , measBytesCopied         :: !Int64     -- bytes copied during GC
  , measMutatorWallSeconds  :: !Double    -- wall-clock on actual work
  , measMutatorCpuSeconds   :: !Double    -- CPU time on actual work
  , measGcWallSeconds       :: !Double    -- wall-clock in GC
  , measGcCpuSeconds        :: !Double    -- CPU time in GC
  }

measure :: Benchmarkable -> Int64 -> IO (Measured, Double)
runBenchmark :: Benchmarkable -> Double -> IO (Vector Measured, Double)
```

### Why Extracted from Criterion?

The extraction was motivated by enabling **alternative analysis frontends**. Before the split, anyone wanting to use criterion's measurement infrastructure had to depend on the entire criterion package -- including its analysis, reporting, charting, and template dependencies. After the split:

- `criterion-measurement` has minimal dependencies (just `base`, `ghc-prim`, `vector`)
- Alternative analysis libraries can import measurement types without pulling in statistics, chart-diagrams, etc.
- Different projects can share the same `Measured` type while using completely different analysis pipelines

This is the strongest prior art for **"data collection separated from analysis"**. The key insight: measurement is a distinct concern from analysis. The `Measured` record is a pure data product -- it makes no analytical claims. Analysis happens downstream, potentially in a completely different library.

## 3. Rust's `statrs` Crate

### Distribution Modeling

The crate separates theoretical distributions from empirical statistics through distinct trait hierarchies:

**Theoretical (distribution parameters -> properties):**
```rust
pub trait Distribution<T: Float> {
    fn mean(&self) -> Option<T> { ... }
    fn variance(&self) -> Option<T> { ... }
    fn std_dev(&self) -> Option<T> { ... }
    fn entropy(&self) -> Option<T> { ... }
    fn skewness(&self) -> Option<T> { ... }
}
```

**Empirical (data -> properties):**
```rust
pub trait Statistics<T> {
    fn min(&self) -> T;
    fn max(&self) -> T;
    fn mean(&self) -> T;
    fn variance(&self) -> T;
    fn std_dev(&self) -> T;
    fn geometric_mean(&self) -> T;
    fn harmonic_mean(&self) -> T;
    fn quadratic_mean(&self) -> T;
    fn population_variance(&self) -> T;
    fn population_std_dev(&self) -> T;
    fn covariance(&self) -> T;
    fn population_covariance(&self) -> T;
    fn abs_min(&self) -> T;
    fn abs_max(&self) -> T;
}

pub trait OrderStatistics<T> {
    fn order_statistic(&mut self, order: usize) -> T;
    fn median(&mut self) -> T;
    fn quantile(&mut self, tau: f64) -> T;   // note: tau is f64 even when T is generic
    fn percentile(&mut self, p: usize) -> T;
    fn lower_quartile(&mut self) -> T;
    fn upper_quartile(&mut self) -> T;
    fn interquartile_range(&mut self) -> T;
    fn ranks(&mut self, tie_breaker: RankTieBreaker) -> Vec<T>;
}
```

### The Generic-vs-Concrete Trade-Off

The traits are defined generically over `T`, but implementations are concrete for `f64`:

```rust
impl<T> Statistics<f64> for T
where T: IntoIterator, T::Item: Borrow<f64>
```

This is the **"generic interface, concrete implementation"** pattern. The trait definition *permits* non-f64 implementations, but the library only provides f64. This means:
- Users of the library always work with `f64`
- Third parties *could* implement `Statistics<f32>` or `Statistics<BigDecimal>`
- The trait boundary is honest about what's provided vs. what's possible

### Descriptive vs. Theoretical Separation

Yes, `statrs` cleanly separates:
- `statrs::statistics` -- traits for empirical computation over data (`Statistics`, `OrderStatistics`)
- `statrs::distribution` -- traits for theoretical distributions (`Continuous`, `Discrete`, `ContinuousCDF`, `DiscreteCDF`)

The same method names (`mean`, `variance`) appear in both contexts with different semantics: `Statistics::mean` computes from data; `Distribution::mean` computes from parameters. The separation is at the module and trait level, not the type level.

### OrderStatistics Requires Mutation

A critical design observation: `OrderStatistics` methods take `&mut self` because they sort in-place. This means quantile/percentile/median are destructive operations -- they modify the data. This is an honest acknowledgment that order statistics require sorted data, and sorting is O(n log n) with side effects.

## 4. Category Theory Perspective

### Sample as Free Commutative Monoid

A `Sample` of observations of type `T` is an element of `FreeCommMon(T)` -- the free commutative monoid on `T`. Concretely, this is a multiset (bag) of `T` values. The monoid operation is multiset union; the identity is the empty multiset.

This is not merely a mathematical curiosity. It has operational consequences:

- **Order irrelevance**: `FreeCommMon(T)` quotients by permutation. A sample `{3, 1, 2}` equals `{1, 2, 3}`. This is correct -- sample statistics should be permutation-invariant.
- **Mergeability**: The monoid operation gives us `sample1 <> sample2 = combinedSample`. Samples are inherently composable.

### Streaming Accumulators as Monoid Homomorphisms

A streaming accumulator is a monoid homomorphism `h: FreeCommMon(T) -> M` where `M` is some monoid of accumulators. The homomorphism law states:

```
h(sample1 <> sample2) = h(sample1) <> h(sample2)
```

This is precisely the property that enables:
1. **Streaming**: Process one observation at a time, `h({x}) <> acc`
2. **Parallel**: Split data, compute `h` on each partition, merge results
3. **Incremental**: Add new data without reprocessing old data

Concrete examples:
- `Sum`: `h({x}) = x`, merge by addition. Homomorphism.
- `Count`: `h({x}) = 1`, merge by addition. Homomorphism.
- `Min/Max`: `h({x}) = x`, merge by min/max. Homomorphism.
- `(Sum, Count)` (for mean): Product of monoids is a monoid. The *accumulator* is a homomorphism; the *mean itself* is a post-processing step `sum/count`.
- `Welford's (n, mean, M2)` (for variance): This is a monoid with Chan et al.'s parallel combination formula. The variance computation is a post-processing step on the accumulator.

### Percentile is NOT a Monoid Homomorphism

`percentile(p)` is a function on `FreeCommMon(T)` that does NOT factor through any finite monoid. Proof sketch: percentile requires knowing the *rank* of every element relative to every other element. For any finite monoid `M`, there exist samples `S1, S2` where `h(S1) = h(S1')` but `percentile(S1 <> S2) != percentile(S1' <> S2)`. The information loss in the homomorphism destroys rank information.

This means: **there is no streaming percentile algorithm with bounded memory**. Any approximate streaming percentile (t-digest, GK sketch, DDSketch) is explicitly trading exactness for bounded space -- they are *approximate* homomorphisms into finite monoids.

### Algebraic Characterization of Decomposable Statistics

**Theorem** (folk, formalized in database theory): A statistic `f: FreeCommMon(T) -> R` is *decomposable* (parallelizable, streamable) if and only if it factors as `f = g . h` where:
- `h: FreeCommMon(T) -> M` is a monoid homomorphism into a finite-dimensional monoid `M`
- `g: M -> R` is a finalization function

The dimension of `M` determines the space complexity of streaming computation.

| Statistic | Decomposable? | Accumulator Monoid | Dimension |
|-----------|---------------|-------------------|-----------|
| sum | Yes | (R, +, 0) | 1 |
| count | Yes | (N, +, 0) | 1 |
| min/max | Yes | (R, min/max, +/-inf) | 1 |
| mean | Yes | (R x N, +, (0,0)) | 2 |
| variance | Yes | (R x R x N, Chan, (0,0,0)) | 3 |
| skewness | Yes | dimension 4 | 4 |
| kurtosis | Yes | dimension 5 | 5 |
| median | No | requires full data | unbounded |
| percentile(p) | No | requires full data | unbounded |
| mode | No | requires full data | unbounded |

### Does This Perspective Help API Design?

**Yes, concretely.** The categorical perspective tells us:

1. **Which statistics compose**: If two statistics are both monoid homomorphisms, their product (computing both simultaneously) is also a monoid homomorphism. This is the theoretical foundation for the "aggregate" pattern.
2. **Where the API boundary falls**: Decomposable statistics can have a streaming/incremental API. Non-decomposable statistics *require* full data access. This is a type-level distinction that should be visible in the API.
3. **Why `Accumulator` and `Sample` are different types**: An `Accumulator` is an element of the target monoid `M`. A `Sample` is an element of `FreeCommMon(T)`. They have different operations and different space characteristics.
4. **The finalization step is not optional**: Mean is not `(Sum, Count)` -- it's the *finalization* of `(Sum, Count)`. The accumulator and the result are different types with different algebraic structures.

## 5. The Foldable/Reducible Connection

### foldMap is the Universal Aggregation Primitive

In Haskell:
```haskell
foldMap :: (Foldable t, Monoid m) => (a -> m) -> t a -> m
```

This says: "given a way to map each observation into a monoid element, and a foldable container of observations, produce the monoidal combination." Every streaming statistic is an instance:

```haskell
-- Sum
foldMap Sum observations

-- Count
foldMap (const (Sum 1)) observations

-- Mean (accumulator only)
foldMap (\x -> (Sum x, Sum 1)) observations

-- Simultaneous min, max, sum, count
foldMap (\x -> (Min x, Max x, Sum x, Sum 1)) observations
```

### Product of Monoids

The product of monoids is a monoid:
```haskell
instance (Monoid a, Monoid b) => Monoid (a, b) where
  mempty = (mempty, mempty)
  (a1, b1) <> (a2, b2) = (a1 <> a2, b1 <> b2)
```

This gives the "aggregate" pattern for free: to compute N statistics simultaneously, take the product of their accumulator monoids. Single traversal, constant memory per statistic.

### Gabriel Gonzalez's Composable Streaming Folds

The `Fold` type abstracts over the monoid:
```haskell
data Fold a b = forall w. (Monoid w) => Fold
  { tally     :: a -> w      -- map observation to accumulator
  , summarize :: w -> b      -- finalize accumulator to result
  }
```

The `Applicative` instance composes folds:
```haskell
average :: Fold Double Double
average = (/) <$> sum <*> genericLength
```

This computes sum and length in a single pass, then divides. The `Applicative` instance pairs the accumulators into a strict product, traverses once, then applies the finalization.

### Mapping to Swift

Swift's `Sequence.reduce(into:_:)` is the operational equivalent of `foldl'`:

```swift
// Haskell: foldl' (\acc x -> acc <> tally x) mempty observations
// Swift:
observations.reduce(into: initialAccumulator) { acc, x in
    acc.combine(with: x)
}
```

A `StatisticsMonoid` protocol could formalize this:

```swift
protocol StatisticsMonoid {
    associatedtype Observation
    associatedtype Result

    static var empty: Self { get }
    mutating func observe(_ value: Observation)
    func merged(with other: Self) -> Self  // monoid operation
    var result: Result { get }             // finalization
}
```

The product-of-monoids pattern maps to tuple conformances or a `Combined<A: StatisticsMonoid, B: StatisticsMonoid>` type. The `observe` method is the composition of the `tally` function with the monoid operation.

## 6. Julia's StatsBase.jl

### Generic Observation Types via Multiple Dispatch

Julia's approach is fundamentally different from Haskell and Rust. There are no trait hierarchies or type class constraints. Instead, functions are generic by default and specialized via multiple dispatch:

```julia
# Any AbstractArray of Real values (including Missing)
function summarystats(a::AbstractArray{T}) where T<:Union{Real,Missing}
    ...
end
```

The `SummaryStats` struct preserves the computed type:

```julia
struct SummaryStats{T<:Union{AbstractFloat,Missing}}
    mean::T
    sd::T
    min::T
    q25::T
    median::T
    q75::T
    max::T
    nobs::Int
    nmiss::Int
end
```

### Type Inference Propagation

Julia infers the output type from the input:
```julia
R = typeof(m)  -- m is the computed mean
SummaryStats{R}(m, stdev, qs..., n, n - ns)
```

If the input is `Float32`, the mean is `Float32`, and the struct is `SummaryStats{Float32}`. If the input contains missing values, the type adapts accordingly. This is the "preserve type information as long as possible" approach in action.

### Multiple Dispatch vs. Trait Constraints

Julia's approach to the constraint problem:
- **No upfront constraint declaration**: Functions just work on whatever types they receive. If `mean(x)` works, so does `summarystats(x)`.
- **Error at call site**: If a type doesn't support the needed operations, you get a method error at the specific failing operation, not an upfront constraint check.
- **Extension by adding methods**: To support a new numeric type, add a `mean` method for it. All downstream functions that call `mean` automatically work.

This is maximally flexible but loses static guarantees. The trade-off is inverted compared to Haskell/Rust/Swift: Julia optimizes for extensibility and expressiveness; Haskell/Rust/Swift optimize for compile-time safety.

## 7. The "Lossy Funnel" Pattern

### Every Ecosystem Funnels to Double

| Language | Mechanism | When |
|----------|-----------|------|
| Haskell | `Sample = Vector Double` | At data ingestion |
| Rust | `impl Statistics<f64> for T` | At trait implementation |
| Go | `float64` everywhere | At function signature |
| Julia | Promotion to `Float64` | At computation time |
| Python/NumPy | `np.float64` default dtype | At array creation |

### Is Premature Narrowing Correct?

**Arguments for early narrowing (Haskell's approach):**
- Eliminates generic constraint propagation (no `Floating + Ord + Unbox + ...` stacks)
- All numerical algorithms are validated for Double precision
- No accidental precision loss from implicit conversions
- Simpler API, fewer type errors, better error messages

**Arguments for late narrowing (preserving type information):**
- `percentile([1, 5, 3, 2, 4], p: 0.5)` should return `Int`, not `Double`
- `min` and `max` should always return `T`, never `Double`
- Only statistics that inherently produce non-integer results (stddev, mean of integers) need floating point
- Better call-site ergonomics: `let midpoint: Int = sample.median` vs. `let midpoint = Int(sample.median)`

### Per-Statistic Type Analysis

| Statistic | Input Type | Natural Result Type | Needs Float? |
|-----------|-----------|-------------------|-------------|
| min, max | T | T | No |
| count | T | Int | No |
| sum | T | T | No |
| percentile | T (Comparable) | T | No |
| median | T (Comparable) | T or Double | Depends |
| range | T (Comparable) | T | No |
| mean | T (Numeric) | Double | Yes (for integers) |
| variance | T (Numeric) | Double | Yes |
| stddev | T (Numeric) | Double | Yes |
| skewness | T (Numeric) | Double | Yes |
| kurtosis | T (Numeric) | Double | Yes |
| correlation | T (Numeric) | Double | Yes |

### The Type-Preserving Approach

The type-preserving approach requires more constraints on `T` but gives better call sites. The key insight is that statistics naturally split into two groups:

1. **Order statistics** (min, max, percentile, median, rank): These only need `Comparable` and return `T`. They operate on the *order structure* of the data.
2. **Moment statistics** (mean, variance, stddev, skewness): These need arithmetic and inherently produce floating-point results. They operate on the *algebraic structure* of the data.

This split is not arbitrary -- it reflects the mathematical structure. Order statistics are functions of the sorted permutation; moment statistics are functions of arithmetic combinations. A well-designed API should make this distinction visible in the type system.

### Recommendation for Swift

The type-preserving approach is feasible in Swift:

```swift
// Order statistics: T -> T (only needs Comparable)
extension Sample where Element: Comparable {
    var min: Element { ... }
    var max: Element { ... }
    func percentile(_ p: Double) -> Element { ... }
    var median: Element { ... }
}

// Moment statistics: T -> Double (needs BinaryFloatingPoint conversion)
extension Sample where Element: BinaryFloatingPoint {
    var mean: Element { ... }
    var variance: Element { ... }
    var standardDeviation: Element { ... }
}

// Integer samples: moment statistics return Double
extension Sample where Element: BinaryInteger {
    var mean: Double { ... }
    var variance: Double { ... }
}
```

This preserves `T` where mathematically sound and narrows to `Double` (or the element's own floating-point type) only where arithmetic demands it. The cost is more overloads; the benefit is type-correct call sites without casts.

## Cross-Cutting Observations

### 1. Separation of Concerns is Universal

Every mature ecosystem separates:
- **Data collection** (measurement, sampling) from **analysis** (statistics, inference)
- **Accumulation** (streaming state) from **finalization** (computing results)
- **Random number generation** from **statistical computation**

### 2. The Monoid Pattern is Universal but Rarely Named

Every ecosystem implements the monoid pattern for streaming statistics, but only Haskell names it explicitly. Rust uses `fold`, Julia uses `reduce`, Go uses loops. The pattern is the same; the abstraction level varies.

### 3. The Generic-vs-Concrete Trade-Off Has a Sweet Spot

- **Too generic** (Rust's trait definition): `Statistics<T>` promises generality that's never delivered -- only `f64` is implemented.
- **Too concrete** (Haskell's `Sample = Vector Double`): Forces conversion at ingestion, losing type information for order statistics.
- **Sweet spot**: Generic for order statistics (only `Comparable`), concrete `Double` for moment statistics. Julia's runtime dispatch approximates this; Swift's constrained extensions can achieve it statically.

### 4. Estimate Parameterization is Underexplored

Haskell's `Estimate e a` (parameterized over error model) is a powerful idea that no other ecosystem replicates. Most libraries return bare `Double` values for confidence intervals. The typed error model prevents mixing a bootstrap confidence interval with a parametric standard error -- a category of bug that is common in practice and invisible without types.

### 5. criterion-measurement is the Design to Study

The extraction of `criterion-measurement` from `criterion` is the cleanest example of the "data collection separated from analysis" pattern. The `Measured` type is a pure data product with no analytical opinions. This is exactly the architecture that enables: one measurement layer, multiple analysis frontends, no coupling between them.
