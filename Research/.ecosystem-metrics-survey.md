### Mutable accumulator

**Type path**: `IO.Blocking.Threads.Aggregate.Mutable` (internal)

```swift
mutating func record(_ durationNs: UInt64)
mutating func record(_ durationNs: Int64)  // guards < 0
func snapshot() -> IO.Blocking.Threads.Aggregate
```

### Notes

- Simple O(1) aggregation: min, max, sum, count.
- No percentiles (P50/P99) -- explicitly deferred to v2, would need HDR histogram or t-digest.
- Average computed by caller: `sumNs / count`.
- Empty state: `count: 0, sumNs: 0, minNs: .max, maxNs: 0`.
- This is the closest thing to a reusable `Measurement` primitive in the ecosystem. Currently coupled to `IO.Blocking.Threads` namespace.

## 3. swift-io: `IO.Blocking.Threads.Counters`

**File**: `/Users/coen/Developer/swift-foundations/swift-io/Sources/IO Blocking Threads/IO.Blocking.Threads.Metrics.swift` (lines 35-73)
**Type path**: `IO.Blocking.Threads.Counters`
**Classification**: Lock-free atomic counter bank
**Visibility**: `internal` (class)

### Architecture

- Reference-backed `Cell` wrapping `Atomic<UInt64>` for each counter.
- Nested accessor structs (`Counter`, `BatchCounter`, `Acceptance`, `Affinity`) for fluent property-chain API.
- All use relaxed atomic ordering (telemetry, eventual consistency acceptable).

### Counter accessors

| Accessor path | Type | Description |
|---------------|------|-------------|
| `counters.enqueued` | `Counter` | Total enqueued |
| `counters.started` | `BatchCounter` | Total started (batch-aware) |
| `counters.completed` | `BatchCounter` | Total completed (batch-aware) |
| `counters.acceptance.promoted` | `Counter` | Promoted from acceptance queue |
| `counters.acceptance.timeout` | `Counter` | Timed out in acceptance queue |
| `counters.failFast` | `Counter` | Rejected with failFast |
| `counters.overloaded` | `Counter` | Rejected with overloaded |
| `counters.cancelled` | `Counter` | Cancelled via Task |
| `counters.affinity.failures` | `Counter` | Affinity routing failures |

### `Counter` vs `BatchCounter`

- `Counter`: `increment()`, `add(_ count: Int)`, `value: UInt64`
- `BatchCounter`: `increment()`, `add(_ count: some Cardinal.Protocol)`, `value: UInt64`
- The distinction is for typed batch increments via `Cardinal.Protocol`.

### Benchmark infrastructure types

| Type | Location | Description |
|------|----------|-------------|

### Key observation

Benchmarks use Swift Testing `@Test` annotations but do NOT use any formal measurement framework. They rely on:
- Wall-clock timing (implicit from test runner)
- `ContinuousClock` for work simulation (`simulateWork()`)
- Counting completed/rejected operations manually
- `IO.Blocking.Threads.Metrics` for verifying counters after runs

There is no shared `Benchmark.Result` or `Measurement.Sample` type -- each benchmark is a standalone scenario with ad-hoc assertions.

## Analysis: Shared Primitive Opportunity

### Pattern summary

| Pattern | Where used | Current implementation |
|---------|------------|----------------------|
| **Monotonic counter** (UInt64, relaxed atomic) | IO.Blocking.Threads.Counters, Pool.Blocking.Metrics, Abandoning.Metrics.Total | Ad-hoc per-site |
| **Gauge** (Int, point-in-time) | IO.Blocking.Threads.Metrics, Pool.Blocking.Metrics, Abandoning.Metrics.Queue, Registry.Metrics | Ad-hoc per-site |
| **High-water mark** (running max) | Pool.Blocking.Metrics.peak | Single site |
| **Latency aggregate** (count/sum/min/max) | IO.Blocking.Threads.Aggregate | Single site, coupled to IO |
| **Advisory atomic gauge** (stale-OK) | IO.Blocking.Threads.Runtime.State.Gauge | Single site |
| **Snapshot** (consistent read under lock) | Both Metrics types | Ad-hoc per-site |

### Could benefit from a shared primitive

1. **`Aggregate` (or `Sample.Aggregate`)** -- The `IO.Blocking.Threads.Aggregate` + `Aggregate.Mutable` pattern is completely domain-independent. It tracks `{count, sumNs, minNs, maxNs}` with O(1) `record()` and `snapshot()`. This could be extracted to a primitives-layer type and reused by:
   - `IO.Blocking.Lane.Abandoning.Metrics` (currently has no latency tracking)
   - `Pool.Blocking.Metrics` (currently has no timing data)
   - Future health checks, scheduling metrics, etc.

2. **Atomic counter bank pattern** -- The `Cell`-backed `Counter`/`BatchCounter` pattern from `IO.Blocking.Threads.Counters` is reusable but currently tied to that namespace. A primitives-layer `Atomic.Counter` with the same `increment()`/`add()`/`value` API could standardize this.

3. **`Metrics` snapshot protocol** -- Both `IO.Blocking.Threads.Metrics` and `Pool.Blocking.Metrics` follow the same pattern: read state under lock, return frozen struct. There is no shared protocol or trait for "provides metrics." A `Metrizable` protocol or witness could unify observability.

### Would NOT benefit from extraction

- **Advisory gauges** -- The `Gauge.Storage` pattern is tightly coupled to the lock/update lifecycle of its specific runtime state. Extracting it would not add value.
- **`DebugSnapshot`** -- Test-only, intentionally coupled to specific internal state.
- **`IO.Executor.Counter`** -- ID generator, not a metric.
- **`Pool.Blocking.Metrics.peak`** -- Too simple to warrant extraction (just a running max).

### Priority recommendation

The single highest-impact extraction is `Aggregate`/`Aggregate.Mutable`. It is:
- Fully domain-independent (only depends on `UInt64`)
- Already production-tested in swift-io
- The only latency tracking primitive in the ecosystem
- Missing from swift-pools (which tracks no timing data at all)
- Missing from the abandoning lane (which has worker/queue metrics but no latency)
- A natural home for future percentile support (HDR histogram, t-digest)
