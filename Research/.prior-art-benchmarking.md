## Executive Summary

Every mature ecosystem eventually separates three concerns: **data collection** (what to measure), **statistical analysis** (how to summarize), and **reporting** (how to present). The critical architectural question is where these boundaries fall and whether the statistical types are generic over the measured quantity or hardwired to duration.

---

## 1. Rust — `criterion` Crate

### Where the Statistical Measurement Type Lives

Separate crate (`criterion`), not in `std`. Rust's stdlib provides `std::time::Duration` and `std::time::Instant` but no sample, statistics, or distribution types. Statistical computation lives entirely in the `criterion` crate (plus its vendored `criterion-stats` internals).

### Generic over Measured Quantity?

**Yes.** The `Measurement` trait is the central abstraction:

```rust
trait Measurement {
    type Intermediate;  // e.g., Instant (start marker)
    type Value;         // e.g., Duration (computed delta)

    fn start(&self) -> Self::Intermediate;
    fn end(&self, i: Self::Intermediate) -> Self::Value;
    fn add(&self, v1: &Self::Value, v2: &Self::Value) -> Self::Value;
    fn zero(&self) -> Self::Value;
    fn to_f64(&self, value: &Self::Value) -> f64;
    fn formatter(&self) -> &dyn ValueFormatter;
}
```

The default implementation is `WallTime` (`Intermediate = Instant`, `Value = Duration`), but the trait admits CPU cycles, GPU counters, or any custom quantity. The `to_f64` method is the escape hatch that collapses any `Value` into the statistical domain.

### Layering

```
measurement.rs      → Data collection (Measurement trait, WallTime)
stats/              → Pure statistical computation (bootstrap, regression)
analysis/           → Applies stats to collected samples
estimate.rs         → Estimate type (point + confidence interval)
report.rs           → Formatting, HTML, CSV output
plot/               → Visualization (gnuplot)
benchmark.rs        → Runner orchestration
benchmark_group.rs  → Grouping with shared config
```

Clean separation: `stats/` has no knowledge of benchmarks. `analysis/` bridges measurement data into statistical types. `report.rs` and `plot/` consume analysis output.

### Type Structure

- **`BenchmarkGroup<M: Measurement>`** — Generic over measurement type
- **`Criterion<M: Measurement>`** — Top-level configuration, also generic
- **Analysis produces**: bootstrap distributions, confidence intervals for mean/median/stddev, linear regression slope, outlier classification (mild/severe via Tukey fences at 1.5x/3x IQR)
- **Comparison**: T-test on bootstrap samples between current and previous runs

### Key Architectural Insight

**The `Measurement` trait decouples "what you measure" from "how you analyze."** The `to_f64` bridge is the critical design: all statistical machinery operates on `f64` regardless of the original quantity. The trait's `Intermediate`/`Value` split cleanly separates the act of capturing a start-marker from computing the measured delta.

## 2. Haskell — `criterion` + `statistics` Packages

### Where the Statistical Measurement Type Lives

**Two separate packages with an explicit extraction boundary:**

| Package | Purpose |
|---------|---------|
| `criterion-measurement` | Measurement types (`Measured`, `Benchmarkable`) — minimal dependencies |
| `criterion` | Runner, analysis, reporting — depends on `statistics` |
| `statistics` | Generic statistical types and functions — completely independent |

The `criterion-measurement` package was explicitly extracted to "enable alternative analysis front-ends." This is the clearest example of the measurement-vs-analysis split.

### Generic over Measured Quantity?

**Partially.** `Measured` is a fixed record (not generic), but `statistics` is fully generic:

```haskell
-- criterion-measurement: Fixed record
data Measured = Measured
  { measTime              :: !Double        -- wall-clock seconds
  , measCpuTime           :: !Double        -- CPU seconds
  , measCycles            :: !Int64         -- rdtsc cycles
  , measIters             :: !Int64         -- loop iterations
  , measAllocated         :: !Int64         -- (GC) bytes allocated
  , measNumGcs            :: !Int64         -- (GC) collections
  , measBytesCopied       :: !Int64         -- (GC) bytes copied
  , measMutatorWallSeconds :: !Double       -- (GC) mutator wall time
  , measMutatorCpuSeconds  :: !Double       -- (GC) mutator CPU time
  , measGcWallSeconds     :: !Double        -- (GC) GC wall time
  , measGcCpuSeconds      :: !Double        -- (GC) GC CPU time
  }

-- statistics: Generic estimate
data Estimate e a = Estimate
  { estPoint :: !a            -- point estimate
  , estError :: !(e a)        -- confidence interval / error
  }

data ConfInt a = ConfInt
  { confIntLDX :: !a          -- lower bound deviation
  , confIntUDX :: !a          -- upper bound deviation
  , confIntCL  :: !(CL Double) -- confidence level
  }
```

The `Estimate` type is parameterized by both the error representation `e` and the value type `a`. This means `Estimate ConfInt Double` and `Estimate NormalErr Double` are different types — the error model is part of the type.

### Layering

```
criterion-measurement     → Measured record, Benchmarkable, measurement I/O
statistics                → Estimate, ConfInt, CL, Sample, distributions, tests
criterion                 → Benchmark tree, SampleAnalysis, Report, KDE, HTML
```

### Type Structure (criterion analysis output)

```haskell
data SampleAnalysis = SampleAnalysis
  { anRegress    :: [Regression]
  , anMean       :: Estimate ConfInt Double
  , anStdDev     :: Estimate ConfInt Double
  , anOutlierVar :: OutlierVariance
  }

data Report = Report
  { reportNumber   :: Int
  , reportName     :: String
  , reportKeys     :: [String]
  , reportMeasured :: Vector Measured    -- raw data
  , reportAnalysis :: SampleAnalysis     -- derived statistics
  , reportOutliers :: Outliers           -- outlier counts
  , reportKDEs     :: [KDE]             -- kernel density estimates
  }

data Outliers = Outliers
  { samplesSeen :: Int64
  , lowSevere   :: Int64
  , lowMild     :: Int64
  , highMild    :: Int64
  , highSevere  :: Int64
  }

data OutlierEffect = Unaffected | Slight | Moderate | Severe
```

### Key Architectural Insight

**Haskell achieves the cleanest separation by making `statistics` a fully independent library.** The `Estimate e a` type is a masterclass in parameterization: the *error model itself* is a type parameter, not just the value. This means the same estimate type works for confidence intervals, normal errors, upper limits, and lower limits. The `criterion-measurement` extraction proves that the measurement record and the statistical analysis are independently useful.

## 3. Go — `testing.B` + `benchstat`

### Where the Statistical Measurement Type Lives

**Split between stdlib and `x/perf`:**

| Location | Purpose |
|----------|---------|
| `testing.B` (stdlib) | Runner, adaptive iteration, raw collection |
| `testing.BenchmarkResult` (stdlib) | Simple result record |
| `golang.org/x/perf/cmd/benchstat` | Statistical comparison (A/B tests) |

### Generic over Measured Quantity?

**No.** The core type is fixed:

```go
type BenchmarkResult struct {
    N         int               // iterations
    T         time.Duration     // total time
    Bytes     int64             // bytes per iteration
    MemAllocs uint64            // total allocations
    MemBytes  uint64            // total bytes allocated
    Extra     map[string]float64 // escape hatch for custom metrics
}
```

The `Extra` map is Go's pragmatic answer to extensibility — `ReportMetric(float64, string)` adds named custom metrics, but they have no type structure.

### Layering

```
testing.B              → Runner (adaptive N, timer control, parallel)
testing.BenchmarkResult → Data record (N, T, Bytes, Mem*, Extra)
benchstat (x/perf)     → Statistical comparison (confidence intervals, A/B)
```

The stdlib intentionally provides only the *runner* and *raw data*. Statistical analysis is delegated to an external tool (`benchstat`) that processes the text output format.

### Key Architectural Insight

**Go separates by process boundary, not by type.** The benchmark runner and the statistical analyzer are different programs communicating through a text format (the Go benchmark format, design doc 14313). This is the most extreme separation of concerns: the stdlib has *zero* statistical types. The tradeoff is that custom metrics lose all type information when they enter the `map[string]float64` escape hatch.

## 4. Java/JVM — JMH + Commons Math + Dropwizard Metrics

### JMH (Java Microbenchmark Harness)

Lives in `org.openjdk.jmh` — separate from JDK, but maintained by OpenJDK. No stdlib benchmarking.

**Result hierarchy:**

```
RunResult
  └── BenchmarkResult (aggregated)
        └── IterationResult[]
              └── Result (abstract)
                    ├── AverageTimeResult
                    ├── ThroughputResult
                    ├── SampleTimeResult
                    └── SingleShotResult
```

Each `Result` carries a `Statistics` object with: `getSum()`, `getMean()`, `getStandardDeviation()`, `getPercentile(double)`, `getN()`, `getConfidenceIntervalAt(double)`.

**Runner architecture:** Annotation processor generates synthetic benchmark code. Fork isolation (default 10 forks), warmup iterations (default 20), measurement iterations (default 20). The generated code handles JIT compilation artifacts, dead code elimination prevention, and blackhole consumption.

### Commons Math (`DescriptiveStatistics` / `SummaryStatistics`)

Completely independent from JMH. Provides two architectural patterns:

| Type | Storage | Use Case |
|------|---------|----------|
| `DescriptiveStatistics` | Stores all values (rolling window) | Small datasets, percentiles |
| `SummaryStatistics` | Streaming (no storage) | Large datasets, online computation |

Both implement `StatisticalSummary` interface. Configurable via pluggable `StorelessUnivariateStatistic` implementations.

### Dropwizard Metrics — Reservoir Pattern

The key contribution is the **reservoir abstraction** for histograms:

| Reservoir | Behavior |
|-----------|----------|
| `ExponentiallyDecayingReservoir` | Default. Biased toward last 5 min. Sample size 1028, alpha 0.015 |
| `SlidingTimeWindowArrayReservoir` | Lossless, fixed time window |
| `SlidingWindowReservoir` | Fixed-count sliding window |
| `UniformReservoir` | Vitter's Algorithm R, uniform sampling |

Percentiles are computed from reservoir contents at read time, not stored directly.

### Key Architectural Insight

**The JVM ecosystem demonstrates three independent layers that never merged:** JMH (benchmark runner) does not use Commons Math (statistical library) does not use Dropwizard (metrics collection). Each reinvents its own `Statistics` type. The Dropwizard reservoir pattern is the most interesting: it separates *which samples to keep* from *what statistics to compute*, recognizing that for long-running systems, the sampling strategy is the critical design decision.

## 5. .NET — BenchmarkDotNet

### Where the Statistical Measurement Type Lives

External library (`BenchmarkDotNet`), not in .NET BCL. Uses `perfolizer` and `pragmastat` as statistical engines.

### Type Structure

```csharp
// Raw measurement point
struct Measurement {
    IterationMode IterationMode;     // Overhead, Workload
    IterationStage IterationStage;   // Pilot, Warmup, Actual
    int LaunchIndex;                 // Process launch index
    int IterationIndex;              // Iteration within launch
    long Operations;                 // Operation count
    double Nanoseconds;              // Total nanoseconds
}

// Computed from Measurement[]
class Statistics {
    Sample Sample;
    double[] OriginalValues;
    int N;
    double Min, Max;
    double Q1, Median, Q3;
    double Mean, StandardDeviation, Variance;
    double Skewness, Kurtosis;
    double StandardError, InterquartileRange;
    double LowerFence, UpperFence;
    double[] LowerOutliers, UpperOutliers, AllOutliers;
    ConfidenceInterval ConfidenceInterval;
    Percentiles Percentiles;  // P0..P100
}
```

### Layering

```
Measurement pipeline:
  Pilot       → choose operation count
  IdleWarmup  → evaluate framework overhead
  IdleTarget  → measure framework overhead
  MainWarmup  → warm up actual method
  MainTarget  → collect actual measurements
  Result = MainTarget - AverageOverhead
```

### Key Architectural Insight

**BenchmarkDotNet's distinguishing feature is explicit overhead subtraction.** The measurement pipeline has dedicated phases for measuring the framework's own overhead, which is then subtracted from the actual measurements. The `Measurement` struct captures the *phase* (via `IterationMode` and `IterationStage`) as part of the data model, making the pipeline stages visible in the type system rather than implicit.

## 6. Observability / Metrics Standards

### OpenTelemetry Data Model

Three metric point types (plus legacy Summary):

| Type | Structure | Aggregability |
|------|-----------|---------------|
| **Histogram** | `count`, `sum`, `min`, `max`, explicit bucket boundaries + counts | Fully aggregable (if boundaries align) |
| **ExponentialHistogram** | `count`, `sum`, `min`, `max`, `scale`, `offset` + dense count arrays, `zero_count` | Fully aggregable (via scale adjustment) |
| **Summary** (legacy) | `count`, `sum`, quantile-value pairs | NOT aggregable |
| **Gauge** | Single value + timestamp | Last-value only |

**Histogram bucket semantics:** Upper bounds inclusive (except +Inf), lower bounds exclusive. A histogram without buckets conveys only sum and count (equivalent to a single (-Inf, +Inf) bucket).

**Exponential histograms** use `base = 2^(2^(-scale))` for bucket boundaries. They support "perfect subsetting" — buckets at one scale map exactly into coarser scales. This is the key innovation for aggregation across heterogeneous sources.

### Prometheus

**Histograms:** Cumulative counters per bucket (le = "less than or equal"). Percentiles computed server-side via `histogram_quantile()`. Native histograms (v2.40+) use dynamic bucket counts in a single time series.

**Summaries:** Client-side quantile computation over a sliding time window. Precise but not aggregable across instances.

**Core tradeoff:** Histograms are *decomposable* (mergeable), summaries are not. This is the fundamental reason histograms are preferred in distributed systems.

### StatsD

Collects raw timer samples until flush interval (default 10s), then computes: min, max, sum, count, mean, percentiles (configurable via `percentThreshold`, default p90). After flush, samples are discarded.

Advanced implementations (Statsite) use Cormode-Muthukrishnan streaming quantile algorithm for memory-bounded percentile estimation with configurable epsilon.

### Key Architectural Insight

**The observability world has converged on a fundamental distinction: decomposable vs. non-decomposable aggregations.** Histograms (bucket counts) are decomposable — you can merge histograms from different sources. Summaries (pre-computed quantiles) are not. This drove OpenTelemetry to deprecate Summary in favor of Histogram. The exponential histogram's scale-based perfect subsetting is the state of the art for balancing resolution against aggregability.

## 7. Academic Foundations

### Georges, Buytaert, Eeckhout (2007) — "Statistically Rigorous Java Performance Evaluation"

**Core methodology:**

1. **Distinguish startup from steady-state.** JIT compilation means early iterations are fundamentally different measurements. Mixing them invalidates all statistics.
2. **Report confidence intervals, not just means.** A mean without a confidence interval is meaningless for comparison.
3. **Use effect sizes, not just p-values.** Statistical significance (p < 0.05) says nothing about practical significance. A 0.1% speedup can be "statistically significant" with enough samples.
4. **ANOVA for multi-factor comparisons.** When comparing more than two configurations, pairwise t-tests inflate Type I error.

**JavaStats tool:** Automated collection framework that enforces the methodology — runs sufficient iterations, separates warmup, computes confidence intervals automatically.

### Kalibera & Jones (2013) — "Rigorous Benchmarking in Reasonable Time"

**Core methodology — Three-level nested random effects model:**

```
Level 3: Builds       (recompilation)
Level 2: Executions   (VM invocations within a build)
Level 1: Iterations   (measurements within an execution)
```

**Key contributions:**

1. **Variance decomposition across levels.** Use nested ANOVA to determine which level contributes most variance. If most variance is between builds, more builds are needed (not more iterations).
2. **Cost-optimal repetition allocation.** Given a time budget, allocate repetitions to the level where variance is highest. The formula balances the cost of each level (builds are expensive, iterations are cheap) against the variance reduction they provide.
3. **Effect size confidence intervals.** Present results as "System A is X% faster than System B, with 95% CI [lo, hi]" — not as p-values.

**Practical impact:** Most benchmarking frameworks only repeat at Level 1 (iterations). Kalibera & Jones showed that Level 2 (executions/VM restarts) and Level 3 (rebuilds) often contribute more variance than Level 1, meaning thousands of iterations within a single execution give false precision.

### Key Architectural Insight

**The academic work establishes that the measurement hierarchy (builds > executions > iterations) is not just a practical concern but a statistical one.** Variance at different levels requires different experimental designs. A benchmarking framework that only supports iteration-level repetition cannot produce statistically sound results for systems with significant execution-level or build-level variance. The framework's data model should capture which level each measurement belongs to.

## Cross-Cutting Analysis

### Separation of Concerns Matrix

| Ecosystem | Data Collection | Statistical Analysis | Reporting | Separation Mechanism |
|-----------|----------------|---------------------|-----------|---------------------|
| Rust/criterion | `Measurement` trait | `stats/` module | `report.rs`, `plot/` | Module boundaries |
| Haskell/criterion | `criterion-measurement` pkg | `statistics` pkg | `criterion` pkg | **Package boundaries** |
| Go | `testing.B` (stdlib) | `benchstat` (x/perf) | `benchstat` text | **Process boundaries** |
| JMH | Runner + codegen | Internal `Statistics` | `ResultFormatFactory` | Class hierarchy |
| BenchmarkDotNet | `Measurement` struct | `perfolizer` engine | `Summary` + exporters | Library boundaries |
| OpenTelemetry | SDK instruments | Backend (server-side) | Visualization layer | **Network boundaries** |

### Genericity Spectrum

| Ecosystem | Measurement Type | Generic? | Escape to Analysis |
|-----------|-----------------|----------|-------------------|
| Rust/criterion | `Measurement::Value` (associated type) | **Fully generic** | `to_f64()` |
| Haskell/statistics | `Estimate e a` | **Fully generic** (value AND error model) | Polymorphic |
| Haskell/criterion | `Measured` record | Fixed record | Field accessors |
| Go | `BenchmarkResult` | Fixed struct | `Extra map[string]float64` |
| BenchmarkDotNet | `Measurement` | Fixed struct (nanoseconds) | `double Nanoseconds` |
| JMH | `Result` hierarchy | Subclass polymorphism | `getScore() -> double` |
| OpenTelemetry | Histogram/Gauge | Fixed schemas | Bucket counts |

### The Universal Pattern

Every ecosystem eventually arrives at the same fundamental pipeline:

```
Collect(raw)  →  Reduce(to numeric)  →  Analyze(statistics)  →  Present(report)
     ^                   ^                      ^                     ^
  domain-specific   lossy bridge         domain-agnostic        format-specific
```

The critical design decision is where the **lossy bridge** lives:
- **Rust**: `to_f64()` method on the trait — explicit, per-measurement-type
- **Haskell**: `Estimate e a` stays polymorphic; analysis functions constrain `a`
- **Go**: `float64` in `Extra` map — implicit, stringly-typed
- **BenchmarkDotNet**: `double Nanoseconds` — fixed at data model level
- **JMH**: `getScore()` abstract method — subclass responsibility

### What the Observability World Teaches

The metrics/observability ecosystem has learned lessons that benchmarking frameworks largely ignore:

1. **Decomposability matters.** If you need to merge results across runs/machines, your data model must support it. Pre-computed percentiles (summaries) cannot be merged. Histograms can.
2. **The sampling strategy is a first-class concern.** Dropwizard's reservoir pattern makes this explicit. Most benchmarking frameworks implicitly use "keep everything" which only works for small sample sets.
3. **Bucket resolution determines percentile accuracy.** OpenTelemetry's exponential histograms show that adaptive bucket strategies outperform fixed boundaries.

### What the Academic Work Teaches

1. **The measurement hierarchy must be in the data model.** Iterations within an execution within a build are statistically nested. Flattening them loses information needed for correct variance estimation.
2. **More iterations is not always better.** If variance comes from the execution level, 10,000 iterations in one execution gives less information than 10 iterations across 10 executions.
3. **Effect size confidence intervals are the correct output.** Not p-values, not bare means, not "X is Y% faster." The interval captures both the magnitude and the uncertainty.
